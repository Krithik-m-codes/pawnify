# Pawnify Security

## Scope and Method

This document covers both subsystems of the Pawnify monorepo as they exist in the working tree today:

- backend/ — a new, partial NestJS API. Exactly 8 feature modules exist under backend/src/modules (auth, cron, customers, health, market-rates, organizations, storage, webhooks), all wired into backend/src/app.module.ts alongside ConfigModule and DatabaseModule as infrastructure-only modules (confirmed by direct read of backend/src/app.module.ts and a directory listing of backend/src/modules). The historical notes at .agents/context.md, .agents/decisions.md, .agents/lessons.md, .agents/schema.md, .agents/tasks.md predate backend/'s existence entirely — none of the five files mentions backend/, NestJS, or a module count of any kind — so there is no actual historical-notes conflict to reconcile on this point; 8 is simply what direct inspection shows.
- web/ — a Next.js App Router application (Prisma, Supabase-hosted Postgres, better-auth, Redux Toolkit Query) that still owns the large majority of business logic: loans, payments, followups, dashboard, reports, settings, staff/admin, profile. None of these have a backend/ counterpart. Only 4 of backend/'s 8 modules are ever called from web/'s UI (customers — search only, market-rates, cron — update-rates only via a manual UI action rather than the scheduled entry declared in web/vercel.json — see Rate Limiting section, organizations — wired but dead/unused in the UI), plus storage via a raw axios call outside the Redux layer. One caveat on the "Next.js 15" characterization commonly used for web/: web/src/app is structured entirely as a modern App-Router/Server-Action codebase, but web/package.json pins `"next": "^9.3.3"` and web/package-lock.json (independently confirmed by direct read in this pass) resolves that pin to the literal, pre-App-Router version 9.3.3. This document does not assert a specific Next.js major version as settled fact given that direct contradiction — see CSRF Considerations for the fuller discussion.

Every claim below states which subsystem(s) it applies to and cites the exact file(s) it was verified against. Claims that could not be verified from the repository are marked "Unknown" rather than guessed. Where the pre-restructure notes at .agents/context.md, .agents/decisions.md, .agents/schema.md conflict with current code, current code is treated as authoritative and the conflict is noted inline. node_modules, dist, .next, coverage, and *.tsbuildinfo were excluded from all analysis; any statement about a third-party library's internal behavior (e.g. the cors npm package's defaults, better-auth's internal cookie/session/routing mechanics) is flagged as such and reflects general documented behavior of that library, not a re-derivation from library source in this pass.

## Executive Summary

Two findings verified directly in this pass are more severe than anything else in this document and should be treated as the top priority regardless of anything else in this document:

1. **RESOLVED (2026-07-11).** backend/'s billing webhook used to be a Stripe integration that never verified the Stripe-Signature header at all — it only checked that STRIPE_WEBHOOK_SECRET was configured, then trusted the raw request body unconditionally, letting any unauthenticated caller POST a forged payload and silently upgrade any organization's billing plan. The integration has since been replaced end-to-end with Dodo Payments (backend/src/modules/webhooks/webhooks.controller.ts, backend/src/modules/webhooks/webhooks.service.ts, backend/src/modules/webhooks/webhooks.repository.ts): the endpoint now binds `@Req() req: RawBodyRequest<Request>` (raw-body capture enabled via `NestFactory.create(AppModule, { rawBody: true })` in backend/src/main.ts) and verifies `req.rawBody` through the official `dodopayments` SDK's `client.webhooks.unwrap()`, a genuine HMAC-SHA256 check per the Standard Webhooks spec, before trusting any payload field — `unwrap()` throws on an invalid/missing signature, turned into a real `401 Invalid webhook signature` response. Verified working end-to-end: a live curl POST with no valid signature headers against the running dev server returned exactly that 401.
2. backend/'s customer search endpoint has no tenant scoping anywhere in its call chain — any authenticated user, from any organization, can enumerate any other organization's customers by name or phone (backend/src/modules/customers/customers.controller.ts, backend/src/modules/customers/customers.service.ts, backend/src/modules/customers/customers.repository.ts).

Beyond these two, the system's overall security posture is: authentication is real and centered on better-auth (web/), which backend/ trusts indirectly by reading the same Postgres session table rather than verifying any cryptographic signature of its own; authorization is a custom, partially-wired RBAC layer with at least one controller (organizations) that is either fully open or permanently locked out depending on the route; there is no rate limiting anywhere in either subsystem; Row-Level Security exists as SQL but is not demonstrably load-bearing for either subsystem's actual database connections; and several pieces of declared configuration (JWT_SECRET, bcryptjs) are dead/vestigial, which is itself a source of false confidence for anyone reading only the .env.example files or package.json.

## Authentication

### web/ — better-auth is the real authentication engine

web/ authenticates users exclusively through the better-auth library, not Supabase Auth. web/src/lib/auth.ts constructs the server-side instance: `betterAuth({ database: prismaAdapter(prisma, { provider: "postgresql" }), trustedOrigins: [...], emailAndPassword: { enabled: true }, session: { expiresIn: 60*60*24, updateAge: 60*60 }, user: { additionalFields: { role, phone, isActive } } })`. Sessions last 24 hours with a 1-hour sliding-update window. The `role` and `isActive` additional fields are declared `input: false`, meaning better-auth's own sign-up payload handling cannot be used by a client to set an elevated role or force-activate an account — every self-registered user is created as STAFF with role/isActive taken from the schema defaults, and role elevation must happen through a separately admin-gated path (web/src/lib/auth.ts). The comment in this file states password hashing is handled internally by better-auth; no custom hashing call was found anywhere in web/src (see Encryption section).

One caveat on reachability, verified directly in this pass: no page or component anywhere in web/src calls `signUp`/`authClient.signUp`, and no signup/register route exists under web/src/app (both grep-confirmed zero matches) — there is no linked front-end registration form. That does not make the underlying endpoint unreachable, though: web/src/app/api/auth/[...all]/route.ts mounts better-auth's entire route set unconditionally via `toNextJsHandler(auth)`, and general better-auth behavior (library-internal, not re-derived from source in this pass) is that enabling `emailAndPassword` exposes its sign-up endpoint at that same mounted path regardless of whether any frontend page links to it. Whether an unauthenticated caller who POSTs directly to that endpoint successfully creates a STAFF-role, active, organization-less account (organizationId has no default and would be left unset) — and what such an account could subsequently do, given that `checkAuth()` (web/src/lib/auth/session.ts) checks only `role` and `isActive` and never `organizationId` — was not fully traced through every Server Action in this pass and is left as Unknown. It is flagged here because it bears on account *creation*, distinct from the role-*elevation* guarantee described above.

web/src/app/api/auth/[...all]/route.ts mounts the instance via `toNextJsHandler(auth)` from `better-auth/next-js` — the standard catch-all integration, exposing sign-in/sign-up/sign-out/session endpoints under /api/auth/*. web/src/lib/auth-client.ts builds the browser client (`createAuthClient` from `better-auth/react`), always targeting the same origin the page was served from (`window.location.origin`) when running in the browser, falling back to `process.env.NEXT_PUBLIC_APP_URL || "http://localhost:3000"` otherwise, and re-exports `signIn`, `signOut`, `signUp`, `useSession`. Server-side session reads go through web/src/lib/auth/session.ts's `getSession()` (`auth.api.getSession({ headers: await headers() })`), with `requireSession()`/`requireAdmin()` redirecting unauthenticated/unauthorized requests, and `checkAuth()`/`checkAdmin()` used inside Server Actions to return a typed error instead of redirecting.

There is no code path anywhere in web/src that calls a Supabase Auth method (`.auth.signInWithPassword`, `.signUp`, `.getUser`, `.onAuthStateChange`, etc.) — Supabase is used for Postgres, storage, and a set of RLS-oriented helper functions that (per the RLS section below) are not actually called from anywhere.

### backend/ — a custom guard that trusts web/'s session table or a genuine Supabase JWT, never a self-issued token

Despite its name, JwtAuthGuard (backend/src/modules/auth/guards/jwt-auth.guard.ts) does not verify a self-signed JSON Web Token. It extracts a credential from, in order: an `Authorization: Bearer` header; a cookie literally named `better-auth.session_token` (parsed cookie object, or a regex match against the raw `Cookie` header) — the exact default cookie name better-auth issues. That credential is handed to `AuthService.validateToken` (backend/src/modules/auth/auth.service.ts), which tries two checks:

1. `AuthRepository.findSessionByToken` (backend/src/modules/auth/auth.repository.ts) runs `SELECT id, "userId", token, "expiresAt" FROM "session" WHERE token = $1 AND "expiresAt" > NOW() LIMIT 1` via a plain node-postgres Pool (backend/src/database/supabase.service.ts) — an exact-match, non-cryptographic lookup against the identical `session` table better-auth's Prisma adapter manages in web/. backend/prisma/schema.prisma and web/prisma/schema.prisma are byte-identical (confirmed via direct diff — zero differences) and both declare `model user`, `session`, `account`, `verification` under a "BETTER AUTH TABLES" comment (line 162), so this works because both subsystems point at one physical database, not because any secret or code is shared.
2. The code falls back to `AuthRepository.verifySupabaseJwtUser` whenever the session-token path did not yield an active, known user — that is, whenever no session row matched the token, *or* a session row matched but no corresponding `user` record was found, *or* the found user's `isActive` flag is false (read directly: the fallback call is unconditional after the session branch, not gated on "no row found" alone). This fallback calls `supabase.auth.getUser(token)` via the `@supabase/supabase-js` SDK — this does perform genuine JWT signature verification, but it is delegated entirely to Supabase's own auth server over the network, using whatever key Supabase's server holds; it never touches backend/'s local `JWT_SECRET`. Because no code path in web/src ever performs a Supabase Auth sign-in, this fallback is not reachable by any traffic web/'s own UI generates today — it exists for some other caller (mobile app, third-party integration, direct API use) that is not present in this repository. Unknown whether any such caller exists.

`JWT_SECRET` is declared in backend/.env.example but is never read anywhere in backend/src — confirmed by grepping the entire backend/src tree, and by reading backend/src/config/configuration.ts, whose exported factory maps `env`, `port`, `apiPrefix`, `database.url`, and `supabase.{url,anonKey,serviceRoleKey}` only, with no `jwtSecret` key at all. backend/package.json's dependency list (`@nestjs/common`, `@nestjs/config`, `@nestjs/core`, `@nestjs/platform-express`, `@nestjs/swagger`, `@prisma/client`, `@supabase/supabase-js`, `@types/pg`, `class-transformer`, `class-validator`, `pg`, `reflect-metadata`, `rxjs`) contains no `passport` or `passport-jwt` package — `JwtAuthGuard` is a fully hand-written `CanActivate`, not NestJS's standard Passport-JWT strategy, despite the name.

auth.controller.ts (backend/src/modules/auth/auth.controller.ts) exposes exactly two routes, both `GET`, both behind `JwtAuthGuard`: `/auth/me` (returns `request.user` via the `CurrentUser` decorator) and `/auth/session` (returns `{ authenticated: true, user }`). There is no login, logout, register, or password-reset endpoint anywhere in backend/ — all credential-based sign-in stays in web/.

### How the two relate

The relationship between web/'s better-auth and backend/'s guard is a shared-database relationship, not a shared-secret or shared-code relationship: backend/ and web/ are two independent npm projects with no imports between them (confirmed — separate package-lock.json files at backend/package-lock.json and web/package-lock.json, no root-level package.json, no turbo.json/nx.json/lerna.json/pnpm-workspace.yaml anywhere, i.e. no monorepo tooling of any kind). backend/'s ConfigModule (backend/src/config/config.module.ts) loads `envFilePath: ['.env', '../web/.env']` — i.e. backend/, when run from its own directory, also reads web/'s .env file one level up — and backend/src/config/configuration.ts's `supabase.url`/`supabase.anonKey` factories fall back to `NEXT_PUBLIC_SUPABASE_URL`/`NEXT_PUBLIC_SUPABASE_ANON_KEY` if the non-prefixed backend-style variables are absent, which only makes sense in light of that shared env-file load path. Both subsystems' Prisma clients ultimately point at one `DATABASE_URL`. Whether backend/'s `DATABASE_URL` is ever actually configured, in a real deployed environment, to point at the same physical database as web/'s is Unknown — the two .env.example files show differently-shaped placeholder values and (per prior infra recon) no docker-compose service currently wires backend/ up at all.

backend/'s own authorization-status check, `AuthService.checkAuthStatus` (backend/src/modules/auth/auth.service.ts), independently re-implements the same rule web/'s `checkAuth()` implements (web/src/lib/auth/session.ts) — down to reusing the identical error strings "Not authenticated. Please sign in." and "Access denied: User account inactive or missing role." — but as two separately maintained implementations, not shared code. One concrete drift: `AuthSessionDto` (backend/src/modules/auth/dto/auth-session.dto.ts) declares an optional `sessionId` field that `AuthController.getSession` never populates, whereas web/'s `checkAuth()` does return a real `sessionId` (`session.session.id`).

## Authorization (RBAC / Roles)

### Role model

The canonical role enum lives in the one Prisma schema shared by both subsystems (backend/prisma/schema.prisma, byte-identical to web/prisma/schema.prisma): `enum Role { PLATFORM_OPERATOR, OWNER, ADMIN, BRANCH_MANAGER, STAFF }` (backend/prisma/schema.prisma, lines 14-20, confirmed by direct read). The `user` model carries `role Role @default(STAFF)`, `isActive Boolean @default(true)`, `organizationId String?` (onDelete: SetNull), and `branchId String?` (onDelete: SetNull) (backend/prisma/schema.prisma, lines 164-194, confirmed by direct read). `AuthUserDto.role` (backend/src/modules/auth/dto/auth-user.dto.ts) types `role` as a plain `string`, not the Prisma enum, so backend/'s auth layer does not statically constrain it.

### web/ enforcement

web/'s own authorization is entirely session-based and string-role-based, implemented in web/src/lib/auth/session.ts: `requireSession()` redirects to `/login` if there is no session or the user lacks `role`/`isActive`; `requireAdmin()` additionally requires `role === "ADMIN"`; `checkAuth()`/`checkAdmin()` are the non-redirecting Server-Action variants used by essentially every `*actions.ts` file backing the RTK Query "action" endpoints. There is no generic multi-role gate in web/ beyond a binary ADMIN check — `BRANCH_MANAGER`, `OWNER`, and `PLATFORM_OPERATOR` are not distinguished by this helper file.

### backend/ enforcement: RolesGuard and @Roles

`Roles` (backend/src/modules/auth/decorators/roles.decorator.ts) is `SetMetadata(ROLES_KEY, roles)` where `ROLES_KEY = 'roles'` — a thin wrapper accepting arbitrary strings, not bound to the `Role` enum. `RolesGuard` (backend/src/modules/auth/guards/roles.guard.ts) reads that metadata via `Reflector.getAllAndOverride`. Its logic, read directly: if no `@Roles(...)` metadata is present on the route, `canActivate` returns `true` immediately — without ever inspecting `request.user`. If roles are declared, it requires `request.user` to already exist (throws 401 "Unauthorized" otherwise), requires `user.isActive` (403 otherwise), and requires `requiredRoles.includes(user.role)` (403 "Access denied: Insufficient privileges." otherwise). `request.user` is populated by exactly one mechanism in the entire codebase: `JwtAuthGuard`'s `canActivate` (backend/src/modules/auth/guards/jwt-auth.guard.ts, line 40). `CurrentUser` (backend/src/modules/auth/decorators/current-user.decorator.ts) is a trivial `request.user` pass-through with no validation of its own.

`AuthModule` is `@Global()` and exports `AuthService`, `JwtAuthGuard`, `RolesGuard` (backend/src/modules/auth/auth.module.ts) — any module can inject these without importing `AuthModule` — but `@Global()` only affects dependency injection availability. Applying a guard to a controller is still an explicit, per-controller `@UseGuards(...)` choice. No global guard exists: backend/src/app.module.ts registers no `APP_GUARD` provider (grep-confirmed zero matches for `APP_GUARD`/`ThrottlerGuard`/`ThrottlerModule` anywhere in backend/src), and backend/src/main.ts's `bootstrap()` wires up only a `ValidationPipe`, `AllExceptionsFilter`, and `LoggingInterceptor` — no auth middleware of any kind.

Guard posture, read directly off every controller in backend/src/modules:

### Confirmed authorization gaps

**OrganizationsController never authenticates (RESOLVED 2026-07-11 — was Critical/P0).** `organizations.controller.ts` (backend/src/modules/organizations/organizations.controller.ts) used to be decorated only with `@UseGuards(RolesGuard)` at the class level — it never applied `JwtAuthGuard` and never injected `@CurrentUser()`. Historically: `getPolicy` (`GET :orgId/policy`) and `getMembers` (`GET :orgId/members`) carried no `@Roles(...)` decorator, so `RolesGuard`'s "no required roles → return true" branch fired immediately, with **zero authentication check of any kind**, letting any caller, given any organization's `id` (a plain, routinely-exposed `cuid`), read that organization's full `LoanPolicy` and its entire team roster; conversely, `updatePolicy`, `inviteMember`, `updateMemberRole`, `removeMember` (all `@Roles('OWNER','ADMIN')`) always saw `request.user === undefined` and threw 401 for every caller. **Fix applied**: the controller now carries `@UseGuards(JwtAuthGuard, RolesGuard)` at the class level, matching customers.controller.ts and storage.controller.ts. Verified working: a curl `GET /api/organizations/:orgId/members` with no credentials now returns 401 instead of leaking data. Not yet added: an explicit check that the authenticated user's own `organizationId` matches the `:orgId` path parameter in the service/repository layer — path parameters should still not be trusted as the tenant boundary on their own, especially given the RLS gaps described later in this document; JwtAuthGuard authenticates the caller, but does not by itself enforce same-tenant access.

**Cross-tenant customer PII leak via unscoped search (Critical — verified directly in this pass, not part of the prior audit batch).** `CustomersController.searchCustomers` (backend/src/modules/customers/customers.controller.ts) is guarded correctly by `JwtAuthGuard` at the class level, and never injects `@CurrentUser()`. It calls `CustomersService.searchCustomers(queryDto.q)` (backend/src/modules/customers/customers.service.ts), which — after a trivial length check (`if (q.length < 2) return []`) — calls `CustomersRepository.searchCustomers(q, 10)` (backend/src/modules/customers/customers.repository.ts). That repository method's Prisma query is:

```
prisma.customer.findMany({
  where: { OR: [{ fullName: { contains: query, mode: 'insensitive' } }, { phone: { contains: query, mode: 'insensitive' } }] },
  select: { id: true, fullName: true, phone: true, city: true },
  orderBy: { fullName: 'asc' },
  take: limit,
})
```

There is no `organizationId` anywhere in this `where` clause, and no `organizationId` (or any other tenant identifier) is ever passed into `searchCustomers` at any layer — the method signature is `searchCustomers(query: string, limit = 10)`, full stop. Because the `Customer` table is not scoped per this query and (per the RLS section below) RLS is not demonstrably enforced for backend/'s own Postgres connection either, **any authenticated user of any organization can enumerate any other organization's customers' full names, phone numbers, and cities** via `GET /api/customers/search?q=<query>`. This requires only a valid session in the system — not a specific organization, not a specific role, not even knowledge of any other tenant's identifiers. This is more severe than the OrganizationsController gap above because it requires no guessing of org IDs and does not depend on a controller-wiring accident being present — it is squarely inside the one endpoint that is correctly authenticated. Fix: add `organizationId` to the `where` clause, sourced from `request.user.organizationId` via `@CurrentUser()` (which the controller does not currently inject at all), not from any client-supplied value.

**Cross-tenant account reassignment in addTeamMember (P2).** `OrganizationsRepository.addTeamMember` (backend/src/modules/organizations/organizations.repository.ts) calls `prisma.user.upsert({ where: { email: dto.email }, update: { role, organizationId, branchId, isActive: true }, ... })`. If `dto.email` already belongs to an existing user row in a *different* organization, the `update` branch unconditionally overwrites that user's `organizationId`, `role`, and `branchId`, and reactivates them (`isActive: true`) — with no check that the matched user is currently unaffiliated or already in the caller's own organization. An OWNER/ADMIN of Org B "inviting" an email that belongs to an existing Org A user would silently move that user into Org B. This is currently unreachable only as an accidental side effect of the OrganizationsController guard gap above (request.user is never populated, so the route 401s) — not a real protection. Fix: reject the upsert with 409/422 if the matched user already belongs to a different organization; only allow it for genuinely unaffiliated users or same-org re-invites.

## JWT Handling

Summarizing what is established above: backend/ declares `JWT_SECRET` (backend/.env.example) but never reads it anywhere in backend/src (confirmed by grep across the whole tree) — it is dead configuration, not wired into any issuance or verification path. No component in either subsystem issues a self-contained, locally-signed JWT for the application's own session mechanism. The two things that could be called "JWT handling" in this system are:

1. Genuine JWT verification that is fully delegated to Supabase's own auth server — `AuthRepository.verifySupabaseJwtUser` (backend/src/modules/auth/auth.repository.ts) calls `supabase.auth.getUser(token)`, which validates a real Supabase-issued access token's signature against Supabase's own keys over the network. backend/ never possesses or checks a signing key for this path itself. As established in the Authentication section, this path is not reachable by any traffic web/'s own login flow generates, since web/ never performs a Supabase Auth sign-in anywhere (zero `.auth.signInWithPassword`/`signUp`/`signIn` calls found in web/src).
2. better-auth's own session tokens (web/src/lib/auth.ts), which — based on the schema (an opaque `token` column on the `session` table, looked up by exact match rather than decoded/verified) — are opaque, database-referenced session identifiers, not self-contained JWTs that carry their own verifiable claims. `BETTER_AUTH_SECRET` (web/.env.example) is passed to the `better-auth` library for its own internal use (cookie signing and/or other internal cryptographic operations); precisely what it protects internally was not verified in this pass, since that lives inside the better-auth library's own source, out of scope for this review. What is confirmed from this repository's own code is that `BETTER_AUTH_SECRET` and `JWT_SECRET` are two entirely separate secrets, read by two entirely separate subsystems, and backend/ never reads `BETTER_AUTH_SECRET` at all (grep-confirmed).

Net effect for anyone reasoning about "JWT security" in this system: there is no independently-signed application JWT to worry about forging, because none is verified anywhere. The actual trust boundary is (a) possession of a live, unexpired row in the shared Postgres `session` table, checked by exact string equality, and (b) for the fallback path only, a real Supabase-verified JWT. Recommendation: either remove `JWT_SECRET` from backend/.env.example to stop misrepresenting the system's actual mechanism, or use it to add a genuine second verification layer.

## Cookies

better-auth issues its session cookie under the default name `better-auth.session_token` (inferred from backend/'s guard expecting exactly that literal name — web/src/lib/auth.ts does not declare an `advanced` block overriding the cookie name, prefix, or attributes). backend/src/modules/auth/guards/jwt-auth.guard.ts reads this cookie via three fallback mechanisms (parsed `cookies` object, then a regex against the raw `Cookie` header) in addition to accepting a `Bearer` header — meaning backend/ accepts the identical browser session cookie web/'s pages use, if it reaches backend/'s origin at all.

No file anywhere in web/src configures explicit cookie attributes (`httpOnly`, `secure`, `sameSite`) for the better-auth session cookie — a repo-wide search for `httpOnly`, `sameSite`, `secure:`, and `advanced` (the better-auth config key that would hold such overrides) in web/src returned zero matches. Cookie security therefore rests entirely on better-auth's own library defaults, which were not independently verified in this pass (would require reading the library's own source under node_modules, out of scope).

Whether the cookie actually reaches backend/'s separate origin in practice is doubtful on the evidence available, for two independent reasons:

- Cookie scope: a cookie set by better-auth on web/'s origin (e.g. `localhost:3000` in local dev, per web/.env.example's `BETTER_AUTH_URL`) is not automatically sent to a different origin (e.g. `localhost:3001`, per web/.env.example's `NEXT_PUBLIC_BACKEND_URL`) unless the cookie's `Domain` attribute is explicitly widened to a shared parent domain. No such configuration (`crossSubDomainCookies` or an equivalent) appears anywhere in web/src/lib/auth.ts.
- CORS credentials: backend/'s only CORS call is `app.enableCors()` with no options object (backend/src/main.ts, line 30). Under the standard, documented behavior of the underlying `cors` npm package, calling it with no options leaves `origin` at its default of `*` and leaves the `credentials` option unset (falsy) — meaning `Access-Control-Allow-Credentials: true` is never sent. Per the CORS specification (general browser behavior, not something specific to this repository), browsers refuse to expose a cross-origin response to client-side JavaScript for a credentialed request (`withCredentials: true`, set on web/src/lib/axiosClient.ts's axios instance) unless the server responds with `Access-Control-Allow-Credentials: true`. Absent that header, the cookie-based path from web/'s browser context to backend/'s API is unlikely to function as intended in a real cross-origin deployment, independent of the `Domain`-scope question above. This is standard CORS/cookie-spec reasoning applied to the configuration found in the repository, not a runtime test performed in this pass.

A second, dead piece of CORS logic exists in web/src/proxy.ts: a function named `proxy` (not `middleware`) that reflects the request's `Origin` header verbatim into `Access-Control-Allow-Origin` while also setting `Access-Control-Allow-Credentials: true` — an open-reflection-plus-credentials combination that would be more permissive (and more risky) than backend/'s bare `enableCors()` if it ever executed. It does not: Next.js only auto-registers middleware from a file literally named `middleware.ts`/`.js`; no such file exists anywhere in the repository (checked both web/middleware.ts and web/src/middleware.ts via glob), and no other file in web/src even contains the substring "proxy" (grep-confirmed — proxy.ts is the only hit for its own name), let alone imports its export. This file has zero effect on any request today, but it documents an apparent intent to allow cross-origin credentialed access broadly, which is worth knowing if anyone "fixes" it by simply renaming it to `middleware.ts` without addressing the reflected-origin problem.

backend/ never sets a cookie of its own anywhere in the controllers/guards reviewed — it is a pure consumer of the cookie better-auth already issued.

### Findings specific to secrets handling

- **Silent anon-key fallback (P2).** `SupabaseService.onModuleInit` (backend/src/database/supabase.service.ts) resolves the client key as `configService.get('supabase.serviceRoleKey') || configService.get('supabase.anonKey') || ''` — if `SUPABASE_SERVICE_ROLE_KEY` is unset or misconfigured, the service silently initializes with the less-privileged anon key instead, logging only a generic "Supabase client initialized" info-level line that does not distinguish which key was actually used. This client is consumed by both `AuthRepository.verifySupabaseJwtUser` (backend/src/modules/auth/auth.repository.ts) and `StorageService.uploadFile` (backend/src/modules/storage/storage.service.ts), so a missing service-role key degrades both silently rather than failing loudly.
- **Vestigial secrets/dependencies give false confidence.** `JWT_SECRET` (backend/.env.example) is unused, as established above. `bcryptjs` and `@types/bcryptjs` are direct dependencies in web/package.json, but a repository-wide, case-insensitive search of web/src found zero references to `bcrypt` anywhere — password hashing is handled entirely inside the better-auth library (per the comment in web/src/lib/auth.ts), making this dependency dead weight that could mislead a future reader into thinking web/ does its own password hashing.
- **NEXT_PUBLIC_APP_URL is undocumented in web/.env.example.** web/src/lib/auth.ts (for better-auth's `trustedOrigins`, itself the documented CSRF mitigation discussed below) and web/src/lib/auth-client.ts (for its non-browser `baseURL` fallback) both read `process.env.NEXT_PUBLIC_APP_URL`, defaulting to `"http://localhost:3000"` if unset (confirmed by direct read of both files). This variable does not appear anywhere among web/.env.example's 6 declared variables. An environment provisioned from the example file alone would silently leave `trustedOrigins`'s own-origin entry pinned at the localhost default in a real deployment unless the variable is set out-of-band — the same class of gap as the DIRECT_URL finding below.
- **.env files are correctly gitignored.** Both backend/.gitignore and web/.gitignore exclude `.env` and `.env.prod` identically (confirmed by direct `diff` of the two files — both blocks are byte-identical); only the `.env.example` placeholder files are tracked. The one actual difference between the two .gitignore files, per that same diff, is that backend/.gitignore additionally ignores `/dist` (backend's NestJS build output directory), which web/.gitignore does not separately list — web/'s equivalent build output, `.next/`, is already covered by a rule present in both files. Separately, per prior infra recon, web/'s Dockerfile does `COPY . .` in its builder stage (confirmed directly in this pass: web/Dockerfile, line 16) with no .dockerignore file anywhere in the repository (checked via repo-wide glob) — a real `.env` file present on disk during a local Docker build would be copied into that intermediate build layer even though it is excluded from git, which is worth closing given how easily intermediate layers/build caches can be exported or cached remotely.
- **DIRECT_URL is undocumented in both example files.** backend/prisma/seed.ts (`new Pool({ connectionString: process.env.DIRECT_URL || process.env.DATABASE_URL })`) and web/prisma.config.ts (`datasource: { url: process.env["DIRECT_URL"] || process.env["DATABASE_URL"] }`, web/'s Prisma CLI config — backend/ has no equivalent `prisma.config.ts` of its own) both resolve a `DIRECT_URL || DATABASE_URL` pattern for direct (non-pooled) connections, but `DIRECT_URL` appears in neither backend/.env.example nor web/.env.example — anyone provisioning a new environment from the example files alone would silently fall back to `DATABASE_URL` for both pooled and direct use, which is not necessarily wrong but is undocumented. (Root docker-compose.yml, discussed further in the RLS section, does set `DIRECT_URL` explicitly for its `app` service — so the variable is genuinely used elsewhere in the repo, just not documented in either .env.example.)

## Rate Limiting

No rate limiting exists anywhere in either subsystem. backend/package.json and web/package.json declare no throttling dependency (no `@nestjs/throttler`, `express-rate-limit`, Upstash ratelimit, or similar). backend/src/main.ts's `bootstrap()` wires up only `ValidationPipe`, `AllExceptionsFilter`, `LoggingInterceptor`, and `app.enableCors()` — no `ThrottlerGuard` or equivalent. A repository-wide, case-insensitive search for `throttle`, `rate-limit`, and `RateLimit` (which also covers both `package-lock.json` files, so it would catch an undeclared transitive dependency too) returns zero matches outside node_modules. web/ has no `middleware.ts` (the dead web/src/proxy.ts notwithstanding), an empty next.config.ts (`{/* config options here */}`, confirmed by direct read), and a vercel.json containing only a cron entry — no edge/firewall rate-limiting configuration anywhere.

That cron entry itself (`"path": "/api/cron/update-rates"`, confirmed in web/vercel.json) does not correspond to any route under web/src/app/api, which — per a directory listing — contains only an `auth/` subdirectory. Vercel Cron Jobs invoke a path on the same deployment the vercel.json belongs to (web/ itself), and neither next.config.ts (a literal empty object, no `rewrites()`) nor vercel.json declares any rewrite from this path to backend/'s separate origin. As configured, this scheduled entry appears disconnected from any live handler in either subsystem. The verified, working call path to backend/'s cron endpoint is different: web/src/components/live-market-ticker.tsx calls `useRefreshMarketRatesMutation` (web/src/lib/redux/api/marketRatesApi.ts, `query: () => ({ url: "/cron/update-rates", method: "POST" })`), which POSTs to backend/'s `/cron/update-rates` via axiosClient when a user clicks the ticker's "Trigger Live Update" control — a manual, UI-triggered action, not an automatic schedule. Note that CronController's only guard, CronAuthGuard, checks for an `Authorization: Bearer <CRON_SECRET>` header (or fails open if `CRON_SECRET` is unset) — and nothing in axiosClient's request interceptor ever attaches that secret (it only ever attaches a bearer token read from `localStorage`'s `access_token`/`better-auth.session_token` keys, per the Cookies/CSRF discussion elsewhere), so this UI button would only succeed in a deployment where `CRON_SECRET` is left unset.

Concretely unthrottled, security-relevant surfaces: the session-token lookup path (`GET /api/auth/me`, `/api/auth/session`, and every `JwtAuthGuard`-protected route) accepts unlimited repeated attempts against the `session` table; the `POST /api/webhooks/dodo-payments` billing webhook (formerly Stripe; see Executive Summary — now signature-verified, but still has no rate limiting) can be hammered without limit; `POST /api/storage/upload` (see File Upload section) can be looped for memory/storage-cost exhaustion; `GET /api/customers/search` runs a Prisma `contains`/`insensitive` scan (see SQL Injection section) with no throttling, a cheap amplification vector layered on top of its cross-tenant scoping bug; and `CronAuthGuard` (backend/src/modules/cron/guards/cron-auth.guard.ts) fails open with no rate limiting behind it if `CRON_SECRET` is ever unset (see below).

## Input Validation

**backend/** uses `class-validator`/`class-transformer` (both direct dependencies, backend/package.json) via a single global `ValidationPipe` registered in `bootstrap()` (backend/src/main.ts): `new ValidationPipe({ whitelist: true, transform: true, forbidNonWhitelisted: true })`. This is applied uniformly to every `@Body()`/`@Query()`-bound DTO class. Coverage is good where DTOs declare real decorators — confirmed directly: `InviteTeamMemberDto`/`UpdateMemberRoleDto` (backend/src/modules/organizations/dto/team-member.dto.ts) use `@IsEmail`, `@IsString`, `@IsEnum`, `@IsNotEmpty`, `@IsOptional`; `UpdateOrganizationPolicyDto` (backend/src/modules/organizations/dto/organization-policy.dto.ts) uses `@IsNumber`/`@Min`/`@Max`/`@IsArray` across every numeric field; `CustomerSearchQueryDto` (backend/src/modules/customers/dto/customer-search-query.dto.ts) validates its one field with `@IsOptional`/`@IsString`.

Two concrete gaps:

- **RESOLVED (2026-07-11).** `StripeWebhookPayloadDto` (backend/src/modules/webhooks/dto/stripe-webhook-payload.dto.ts) used to declare `type?: string` and a nested `data?.object?.metadata?: { organizationId?: string; planId?: string }` shape with only `@ApiPropertyOptional` (Swagger metadata) — zero `class-validator` decorators anywhere in the file — which was a secondary concern next to the complete absence of signature verification on the same endpoint. The DTO no longer exists: the webhook was migrated from Stripe to Dodo Payments, and the endpoint now binds `@Req() req: RawBodyRequest<Request>` instead of a `@Body()`-bound DTO, verifying `req.rawBody` via the `dodopayments` SDK's `client.webhooks.unwrap()` (real HMAC-SHA256 signature check) before any field is trusted — see Executive Summary.
- `StorageController.uploadFile`'s bucket parameter (backend/src/modules/storage/storage.controller.ts) is extracted via `@Body('bucket') bucket?: string` — a raw single-property extraction with no DTO class attached, which NestJS's `ValidationPipe` does not run `class-validator` against at all. This value flows unchecked into `StorageService.uploadFile` and straight into `supabase.storage.from(bucketName)` (backend/src/modules/storage/storage.service.ts) with no allow-list.

**web/** does not use `class-validator` (it is a Next.js app, not NestJS) — instead it defines Zod schemas as, per its own header comment, "the source of truth for validation," re-validated server-side regardless of client-side use. Confirmed by reading web/src/lib/validation/customer.ts: `customerCreateSchema` validates name length (2-100 chars), an India-specific 10-digit mobile regex (`/^[6-9]\d{9}$/`), optional email, required address fields, and a 6-digit pincode regex; `kycDocumentSchema` uses a Zod `superRefine` to apply format-specific regex validation per `docType` (PAN, Aadhaar, Voter ID, Passport, Driving License patterns) and additionally exports a `maskDocNumber` helper that redacts all but the last 4 characters for UI display. Equivalent schema files exist for loans and payments (web/src/lib/validation/loan.ts, web/src/lib/validation/payment.ts; confirmed to exist, not read in full in this pass). This is a reasonable, India-market-specific validation design, applied consistently at the point closest to where each domain object is created.

## XSS Considerations

React/Next.js escapes all JSX-interpolated values by default in both the App Router pages and any component tree in web/. A repository-wide search for `dangerouslySetInnerHTML` across web/src returned zero matches — no component anywhere bypasses React's default escaping.

The residual XSS-adjacent risk is a stored-content concern tied to the File Upload gaps below: `document-uploader.tsx` (confirmed by direct read) renders uploaded document URLs both as `<img src={doc.url}>` previews (list and preview-modal views) and as `<a href={doc.url} target="_blank" rel="noopener noreferrer">` links. Its own client-side `accept` list (`".pdf,.png,.jpg,.jpeg,.webp,.gif,.svg,.doc,.docx,.xls,.xlsx,.txt,image/*"`) explicitly includes `.svg`. Because `StorageService.uploadFile` (backend/src/modules/storage/storage.service.ts) never validates `file.mimetype` against an allow-list and forwards the client-supplied MIME type straight through as the stored object's `contentType`, an attacker (or simply a normal user, since `.svg` is an accepted type in the UI itself) could upload an SVG embedding `<script>`; if that URL is later opened directly in a new tab (as the `target="_blank"` anchors do), the SVG would execute in the context of the Supabase Storage origin, not web/'s or backend/'s origin — this limits blast radius to whatever that storage origin can do (typically read cookies/storage scoped to that origin, not Pawnify's own), but is still a real content-injection primitive worth closing at the upload-validation layer.

No Content-Security-Policy, `X-Frame-Options`, or other security-header configuration exists anywhere: web/next.config.ts's exported config is a literal empty object (`{/* config options here */}`, confirmed by direct read), and no `helmet` (or equivalent) package appears in either package.json (grep-confirmed for `helmet` across both package.json files). This means the application relies solely on React's default output escaping for XSS defense-in-depth, with no CSP as a second line of defense if that escaping were ever bypassed by a future code change.

## CSRF Considerations

No CSRF middleware, token library, or double-submit-cookie pattern exists anywhere in the repository — a case-insensitive, repository-wide search for `csrf`/`xsrf` returned zero matches in either subsystem's tracked source.

For **web/**'s own better-auth routes (`/api/auth/*`), better-auth's documented mitigation for this class of attack is checking the request's `Origin`/`Referer` header against a configured allow-list — web/src/lib/auth.ts does configure `trustedOrigins` (the app's own `NEXT_PUBLIC_APP_URL`, a hardcoded production URL `"https://pawnify-three.vercel.app"`, and the Vercel preview URL via `process.env.VERCEL_URL` when present — all confirmed by direct read). Whether this fully covers CSRF for better-auth's endpoints, and what cookie `SameSite` default better-auth applies underneath that check, was not independently verified in this pass (would require reading the library's internals, out of scope). For the 31-of-40 RTK Query endpoints that call Next.js Server Actions directly in-process (web/src/lib/redux/api/baseApi.ts's `hybridAxiosBaseQuery`, `action` branch — this exact 31/40 split was independently recomputed in this pass by counting `builder.query`/`builder.mutation` definitions and their `action:`-vs-`url:` query shape across all 10 files under web/src/lib/redux/api/), Next.js's own Server Action implementation includes built-in Origin-header CSRF checking in the framework versions that support Server Actions at all. This is where a genuine "Unknown" must be flagged rather than assumed: web/package.json pins `"next": "^9.3.3"`, and this was independently re-confirmed in this pass by reading web/package-lock.json directly, whose `node_modules/next` entry resolves to the literal version `9.3.3` — a 2020-era release that predates the App Router and Server Actions entirely, while web/src/app is written entirely as a modern App-Router/Server-Action codebase (confirmed further by 15 files under web/src containing a `"use server"` directive). Notably, web/package.json's own devDependencies pin `"eslint-config-next": "16.2.10"` — a version number wildly inconsistent with `"next": "^9.3.3"` in the same file, circumstantial evidence that the `next` version pin itself is likely a mistake rather than an intentional downgrade, though this remains unconfirmed. Which Next.js version's CSRF behavior (if any) would actually apply to a real running instance of this code cannot be determined from the repository alone.

For **backend/**, the relevant question is whether any state-changing route can be triggered by a forged cross-site request riding on the browser's automatically-attached `better-auth.session_token` cookie. Two things limit this in practice, neither independently confirmed as a deliberate defense: backend/'s `enableCors()` does not enable `credentials`, which — per the CORS/cookie reasoning in the Cookies section above — likely prevents the cookie-based path from functioning cross-origin at all in a real browser; and CORS itself is not a CSRF defense regardless (CORS restricts which origins JavaScript can read a response from, not whether a browser will send a cookie-bearing request to begin with — a `<form>` POST or simple cross-site `fetch` with `no-cors` mode can still reach the server and trigger a state change even where CORS would block reading the response). No explicit anti-CSRF token check exists on any backend/ route. Given the CORS/cookie ambiguity above, whether this is currently exploitable end-to-end is Unknown; it should not be relied upon as a defense.

## SQL Injection Considerations

The overwhelming majority of database access in both subsystems goes through Prisma's query builder (`prisma.<model>.findMany/findUnique/create/update/upsert(...)`), which parameterizes all inputs — no string-concatenated SQL was found anywhere via a repository-wide search for the common raw-query APIs. The specific occurrences of raw SQL, all checked directly:

- backend/src/modules/auth/auth.repository.ts's `findSessionByToken` and `findUserById` use parameterized placeholders (`WHERE token = $1`, `WHERE id = $1`) against a plain `pg.Pool` (backend/src/database/supabase.service.ts) — safely parameterized.
- backend/src/modules/health/health.repository.ts runs `this.prisma.$queryRaw\`SELECT 1\`` — a literal, argument-free tagged-template query; no injection surface at all.
- web/src/lib/supabase/server.ts's `runWithTenantContext` runs `tx.$executeRawUnsafe('SELECT set_config(\'app.current_organization_id\', $1, true)', organizationId)` — despite the `Unsafe` suffix in Prisma's own naming (which refers to the query *string* not being checked, unlike the tagged-template form), the value itself is passed as a bound parameter ($1) to the underlying driver, not string-concatenated into the SQL text — this is safely parameterized. (This helper is, however, never called from anywhere else in web/src — confirmed by a repository-wide search for its name finding only its own defining file — see RLS section.)

`CustomersRepository.searchCustomers`'s `contains`/`mode: 'insensitive'` filter (backend/src/modules/customers/customers.repository.ts) is Prisma-generated `ILIKE`-equivalent SQL with the search term passed as a bound parameter — not vulnerable to injection, though (as covered in Authorization) it is vulnerable to cross-tenant data exposure, and (as covered in Rate Limiting) it has no throttling against repeated/expensive pattern scans.

No instance of raw string interpolation of user input into a SQL string (e.g. `` `SELECT * FROM x WHERE y = '${input}'` ``) was found anywhere in backend/ or web/.

## File Upload Security (Storage Module)

`StorageController.uploadFile` (backend/src/modules/storage/storage.controller.ts) is defined with `@UseInterceptors(FileInterceptor('file'))` and no options object — no `limits.fileSize`, no `fileFilter`. Combined with `StorageService.uploadFile` (backend/src/modules/storage/storage.service.ts), which only checks that `file`/`file.buffer` exist before proceeding, this endpoint (guarded solely by `JwtAuthGuard` — any authenticated user of any role, from any organization, may call it) has:

- **No size cap.** Multer's default in-memory storage engine buffers the entire upload before any check runs (general, documented Multer/`@nestjs/platform-express` behavior); nothing rejects an arbitrarily large file, enabling memory-exhaustion or storage-quota-exhaustion abuse.
- **No MIME/extension allow-list.** `file.mimetype` (fully client-controlled) is forwarded unchanged as the stored object's `contentType` (backend/src/modules/storage/storage.service.ts, line 40) with no validation against the intended document types (KYC scans, collateral photos). Only the filename is sanitized, via `file.originalname.replace(/[^a-zA-Z0-9.-]/g, '_')` — this prevents path-traversal-via-filename but does nothing about file *content* or declared MIME type.
- **Unvalidated bucket parameter.** `bucket` is accepted as a free-text `@Body('bucket') bucket?: string` with no DTO/allow-list (see Input Validation section) and forwarded verbatim to `supabase.storage.from(bucketName)` — any bucket name reachable by the configured Supabase key can be written to, not just the intended `pawnify-docs` default.
- **Service-role-key client.** `SupabaseService` (backend/src/database/supabase.service.ts) builds its Supabase client from the service-role key (falling back to the anon key only if the service-role key is absent — see Secrets Management), which bypasses per-bucket Supabase Storage policies by design.
- **Public URLs, not signed URLs.** `getPublicUrl` is used (backend/src/modules/storage/storage.service.ts, line 51), not a time-limited signed URL, for content that includes KYC documents.
- **Error messages leak driver detail to the client.** Both catch blocks in `StorageService.uploadFile` throw an `HttpException` whose body is the raw `error.message`/caught exception message (backend/src/modules/storage/storage.service.ts, lines 46 and 63) rather than a generic message — a minor information-disclosure pattern that mirrors the same issue independently found in the health module (`HealthService.getHealthStatus`, backend/src/modules/health/health.service.ts, which returns `error: error.message` on a database-connectivity failure to an unauthenticated caller).

On the **web/** side, the live upload UI (`document-uploader.tsx`, confirmed by grep to be rendered from web/src/app/(app)/customers/new/page.tsx, web/src/app/(app)/loans/new/page.tsx, and web/src/app/(app)/profile/profile-client.tsx) does apply an `accept` file-type list and a default 15 MB size cap (`maxSizeMB = 15`) — but both are plain client-side JavaScript checks in `handleFileSelect`, trivially bypassed by calling `POST /api/storage/upload` directly with any tool, since none of these constraints are re-enforced server-side.

Separately, web/ ships two additional, currently-dead Supabase upload helpers with the same class of risk if ever wired up: `uploadToSupabaseBucket` (web/src/lib/supabase-storage.ts) resolves its key as `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY || SUPABASE_ANON_KEY || SUPABASE_SERVICE_ROLE_KEY` and accepts unvalidated `bucketName`/`contentType` arguments; `createAdminSupabaseClient` (web/src/lib/supabase/server.ts) is explicitly service-role-backed. A repository-wide search confirms neither is imported or called from anywhere else in web/src today (both found only in their own defining files).

## Row-Level Security as the Tenant-Isolation Boundary

backend/sql/rls_policies.sql is the one artifact in the repository explicitly framed as tenant-isolation security enforcement, and it is backend/-only (no web/ counterpart) even though, because both subsystems share one physical database via `DATABASE_URL`, any policy it actually creates would govern both subsystems' connections equally once applied.

**What the SQL defines (confirmed by direct read of backend/sql/rls_policies.sql).** `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` on exactly 12 tables: `Organization`, `Branch`, `LoanPolicy`, `DocumentType`, `Customer`, `KycDocument`, `Loan`, `LoanItem`, `LoanCharge`, `Payment`, `LedgerEntry`, `FollowUp`. A helper function, `current_user_organization_id()`, defined as `COALESCE(NULLIF(current_setting('app.current_organization_id', true), ''), (SELECT "organizationId" FROM "user" WHERE id = auth.uid() LIMIT 1))` — i.e. it first tries a session-local Postgres setting, then falls back to looking up the caller's organization via Supabase's `auth.uid()` (populated from a Supabase-issued JWT's claims via PostgREST). `Organization` gets exactly one policy, `FOR SELECT USING (id = current_user_organization_id())` — no `INSERT`/`UPDATE`/`DELETE` policy exists for `Organization` at all. The remaining 11 tables each get one `FOR ALL` policy: `USING ("organizationId" = current_user_organization_id()) WITH CHECK (...)`.

**Gaps in the policy set itself, independent of whether it is enforced.** The `user` table carries the identical tenant-scoping shape as every protected table (`organizationId`, `branchId` columns — backend/prisma/schema.prisma) yet is entirely absent from the `ENABLE ROW LEVEL SECURITY` list and has no policy of any kind — under standard Postgres semantics, a table with RLS not enabled is fully readable/writable (subject only to ordinary SQL grants) by any role with table privileges. `session`, `account`, `verification` (holding session tokens and password-hash columns) have no `organizationId` column and are correctly out of scope for this organization-keyed scheme, but are also completely unprotected by RLS and are not scoped to `userId` at the policy level either. No `FORCE ROW LEVEL SECURITY` statement appears anywhere, so — under general Postgres behavior — whichever role owns these tables continues to bypass every policy regardless of its content. `Organization`'s SELECT-only policy means that `WebhooksRepository.updateOrganizationPlan` (backend/src/modules/webhooks/webhooks.repository.ts, `prisma.organization.updateMany({ where: { id: organizationId }, data: { billingPlan: newPlan } })`) is a real, currently-live write path against `Organization` that no policy authorizes — it can only be working today because the connecting role bypasses RLS outright.

**Whether it is actually enforced for either subsystem's own connections — the more important question.** The session-variable branch of `current_user_organization_id()` depends on `set_config('app.current_organization_id', ...)` having been run on the same connection beforehand. A repository-wide search finds exactly one place this is ever called: `runWithTenantContext` (web/src/lib/supabase/server.ts). A further search for callers of `runWithTenantContext` anywhere in web/src outside its own defining file returns zero matches — by static analysis, this helper is never invoked. No occurrence of `set_config` or `current_organization_id` exists anywhere in backend/src at all. The `auth.uid()` fallback branch requires a Supabase-issued JWT's claims to have reached Postgres via PostgREST — but neither subsystem's direct database access goes through PostgREST as an authenticated role: backend/'s `PrismaService` (backend/src/database/prisma.service.ts) is a bare `PrismaClient` subclass with no session-scoping logic at all, and its `onModuleInit` only logs a warning (rather than throwing) if `$connect()` fails, suggesting the connection mechanism itself was not fully finalized; backend/'s `SupabaseService` uses a plain `pg.Pool` for its raw-SQL path and a Supabase client authenticated with the service-role key (not a per-request user JWT) for its Storage/Auth-verification path. web/'s own direct Prisma access (web/src/lib/db.ts) is likewise a plain `pg.Pool`-backed `PrismaClient` (via `@prisma/adapter-pg`), not a PostgREST-routed, per-user-JWT-authenticated connection.

**Practical conclusion.** In this codebase, as it stands, tenant isolation is enforced almost entirely by explicit, hand-written `organizationId` filters in application code — confirmed directly in web/src/lib/services/customers.ts (`prisma.customer.create({ data: { organizationId: data.organizationId, ... } })` and equivalent `where` filters elsewhere) and in most of backend/'s repository methods (e.g. `OrganizationsRepository.updateMemberRole`/`removeTeamMember`, both scoping by `{ id, organizationId }`) — not by the RLS policies in backend/sql/rls_policies.sql, which have no demonstrated mechanism to actually engage for either subsystem's own connections. Where that manual discipline lapses, as in `CustomersRepository.searchCustomers` (see Authorization section), there is no database-level backstop to catch the omission — RLS's presence in the codebase should not be read as an active safety net today. Whether backend/sql/rls_policies.sql has ever actually been applied to any live database is itself Unknown: neither backend/prisma nor web/prisma has a `migrations` directory (schema changes go through `prisma db push` — confirmed by directory listing showing no `migrations` subdirectory under either), and no script in either package.json executes this SQL file (confirmed by reading both files' `scripts` blocks in full) — it must be run manually (e.g. via `psql` or the Supabase SQL editor) and there is no evidence in the repository that this has happened.

Separately, root docker-compose.yml's `db` service expects to mount this file from `./sql/rls_policies.sql` (repository root) at container init, but no such path exists on disk — the file lives only at backend/sql/rls_policies.sql — so this automatic-provisioning path is currently broken (confirmed: `./sql` does not exist at the repository root at all, only `./backend/sql`). A second, independent break sits right next to it in the same docker-compose.yml: the `app` service builds with `context: .` (the repository root) and `dockerfile: Dockerfile`, i.e. it looks for a Dockerfile at the repository root — which also does not exist (confirmed by direct check). Only web/Dockerfile exists, and that file's own early-stage `COPY` instructions (`COPY package.json package-lock.json* ./`, `COPY prisma ./prisma`) assume a build context rooted at web/ itself, not the repository root. As currently configured, `docker compose build` for the `app` service would fail outright to even locate its Dockerfile, independent of the RLS SQL-mount issue described above.

**Conflict with prior project notes.** .agents/decisions.md (DEC-001) states RLS is "load-bearing" and "verified via src/__tests__/rls-isolation.test.ts." Reading web/src/__tests__/rls-isolation.test.ts directly shows two Vitest cases that filter plain in-memory JavaScript arrays (e.g. `records.filter((r) => r.organizationId === "org-a")`), and neither ever connects to Postgres, calls Prisma or Supabase, or exercises backend/sql/rls_policies.sql's actual SQL policies. Per this document's own rule that current code wins over stale historical notes, this is flagged explicitly: the "verified" claim in decisions.md does not match what the cited test file actually does, and should not be relied upon as evidence that RLS enforcement has ever been tested end-to-end.

Ordered by severity, grounded in the findings above. "Critical" items are directly exploitable today by a party with minimal or no privilege; lower items are either harder to reach, lower-impact, or contingent on other misconfiguration.
