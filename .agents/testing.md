# Pawnify Testing

Scope: this document covers testing across both subsystems in this monorepo — web/ (mature Next.js app, Vitest) and backend/ (partial NestJS API, Jest). Every claim below states which subsystem(s) it applies to. Facts were verified by directly reading the cited source files and, where noted, by actually executing both test suites in this environment on 2026-07-11 — execution results are called out explicitly since they are more authoritative than static inference and can go stale if dependencies change. Historical planning notes under .agents/ describe a pre-restructure single-app repository and are treated as background only; where they conflict with current code, current code wins (see "Spec-vs-code drift" below).

## 1. Test suite topology

Two entirely separate, non-integrated test setups exist. There is no CI of any kind (no .github directory anywhere in the repo) and no monorepo test orchestration (no turbo.json/nx.json/pnpm-workspace.yaml/lerna.json) — nothing invokes both suites together, and nothing runs either suite automatically on push or PR.

### 3.2 The e2e test: what it covers, and why it currently fails

backend/test/app.e2e-spec.ts boots the full AppModule via NestJS's TestingModule and makes exactly one HTTP assertion: GET / expecting HTTP 200 with body exactly "Hello World!" (backend/test/app.e2e-spec.ts, lines 11-24). This is the unmodified Nest CLI scaffold default. It never issues a request to any route belonging to the 8 real modules — not /auth/*, /customers/*, /market-rates/*, /organizations/*, /storage/*, /webhooks/*, /cron/*, nor health's own real endpoint (backend/src/modules/health/health.controller.ts). These are the controllers' own relative paths — the routes actually reachable on the running server carry the global "api" prefix applied via `app.setGlobalPrefix(apiPrefix)` (backend/src/main.ts, line 17), where apiPrefix defaults to 'api' (backend/src/config/configuration.ts, line 4) and is also explicitly set via API_PREFIX=api in backend/.env, backend/.env.example, and backend/.env.prod — e.g. the real health check is GET /api/health, not GET /health.

Beyond being narrowly scoped, this test currently does not pass. Confirmed by executing npx jest --config ./test/jest-e2e.json from backend/ on 2026-07-11, for two independent reasons:

1. **It throws during setup, before the HTTP request is ever made.** TestingModule.compile() fails with `PrismaClientInitializationError: PrismaClient needs to be constructed with a non-empty, valid PrismaClientOptions`, originating from `new PrismaService` (backend/src/database/prisma.service.ts, line 5 — the class extends PrismaClient with no constructor override, so the implicit constructor calls super() with zero arguments). Comparing directly against web/'s equivalent module narrows this down further than a version mismatch alone: neither backend/prisma/schema.prisma nor web/prisma/schema.prisma declares a `url` in its `datasource db { ... }` block — both just say `provider = "postgresql"`. web/'s runtime client (web/src/lib/db.ts, lines 28-34) supplies the missing connection info explicitly at construction time via `new PrismaClient({ adapter })`, where `adapter = new PrismaPg(pool)` and `pool` is a pg.Pool built from `process.env.DATABASE_URL` — using the "@prisma/adapter-pg" package, which is a dependency in web/package.json but is absent entirely from backend/package.json (checked directly: backend's only Postgres-related dependencies are "@prisma/client" and the bare "pg" driver, and "pg" is used only by backend/src/database/supabase.service.ts's separate connection pool, never wired to PrismaService). backend/src/database/prisma.service.ts supplies neither a `url` nor an adapter, so it has no configured way to obtain a connection string, independent of whether DATABASE_URL itself is set in the environment. This is a more direct explanation than the cross-major-version split the two Prisma packages are also in: backend/package.json declares "@prisma/client": "^7.8.0" alongside "prisma": "^6.19.3" (the CLI/generator) in devDependencies — but web/package.json declares the exact same split and its Prisma-backed code runs without this error, which weighs against the version split being the operative cause here. This document does not modify source, so the adapter/url gap is reported as strong, directly-observed comparative evidence rather than a confirmed fix. Because compilation never finishes, the `app` variable is never assigned, and the test's own afterEach then throws a second, cascading `TypeError: Cannot read properties of undefined (reading 'close')` at backend/test/app.e2e-spec.ts, line 27.
2. **Independently of the Prisma error, the assertion has no controller to satisfy in current source.** backend/src/app.module.ts declares no controllers array and imports no AppController; there is no app.controller.ts anywhere under backend/src (confirmed by a case-insensitive repo search); none of the 8 real module controllers registers an empty/root path — each uses a named prefix (`@Controller('auth')`, `@Controller('cron')`, `@Controller('customers')`, `@Controller('health')`, `@Controller('market-rates')`, `@Controller('organizations')`, `@Controller('storage')`, `@Controller('webhooks')`); and the literal string "Hello World" appears nowhere in backend/src production code — only in the test's own expectation string.

Net effect: backend/'s test suite, unit and e2e combined, currently contributes 0 passing tests and 0 verified regression protection for any of its 8 real modules. Even setting aside scope, there is presently no green baseline to build on in backend/.

## 7. Spec-vs-code drift affecting testing claims

**THIRTY_360 day-count convention (web/ only).** The Prisma schema declares all three conventions (web/prisma/schema.prisma, lines 39-42: `enum DayCountConvention { ACTUAL_365 ACTUAL_360 THIRTY_360 }`, defaulted to ACTUAL_365 at line 138) and the onboarding UI offers THIRTY_360 as a selectable `<option>` labeled "30 / 360 (Fixed Month)" bound to a live form field (web/src/app/(app)/onboarding/onboarding-wizard-client.tsx, line 356, confirmed by direct read this session). But computeDailyInterest implements only a two-way branch, confirmed by direct read this session: `const divisor = dayCountConvention === "ACTUAL_360" ? new Decimal(360) : new Decimal(365);` (web/src/lib/services/interest.ts, line 36) — any value other than the literal "ACTUAL_360", including "THIRTY_360", silently falls through to the 365 divisor, identical to ACTUAL_365, and elapsed days are always true calendar days (`differenceInCalendarDays`, line 61), never a 30-day-per-month count. No test in the repo passes or asserts anything about THIRTY_360 — interest.test.ts never passes a convention argument at all, and jurisdiction.test.ts's one day-count test only compares ACTUAL_365 against ACTUAL_360. This is simultaneously a business-logic discrepancy and a coverage gap: an organization that selects "30 / 360 (Fixed Month)" during onboarding silently receives ACTUAL_365 behavior.

**Stale test-coverage claims in .agents/ historical notes (applies to web/'s pre-restructure code, since these notes predate backend/'s existence).** Two entries directly contradict current code:

- .agents/decisions.md, DEC-001 (Row-Level Security): "Status: RESOLVED & EXECUTED (Phase 1)" ... "Verification: Tested via src/__tests__/rls-isolation.test.ts." Per project convention, current code wins: both notes should be read as stale intent, not as evidence of actual coverage.

**Stale RLS-enforcement design claim (applies to web/'s current code vs. pre-restructure intent).** .agents/context.md states the intended data-access design: "Load-Bearing RLS: Row-Level Security enforced at Postgres level. Tenant CRUD uses Supabase client SDK carrying authenticated JWTs; Prisma reserved for schema migrations and service-role cross-tenant admin operations" (.agents/context.md, line 12). Current code does not follow this: every web/ service file examined in this document (payments.ts, loans.ts, customers.ts, dashboard.ts, valuation.ts, billing.ts, settings.ts) queries exclusively through the plain `prisma` singleton (web/src/lib/db.ts); none of them — nor anything else in the repo — calls createTenantSupabaseClient or createAdminSupabaseClient (web/src/lib/supabase/server.ts), which, as noted in section 6, have zero call sites anywhere. Current code wins: the actual data-access pattern is Prisma-only, not the JWT-carrying-Supabase-client pattern the note describes, which is directly relevant to whether the RLS policies in backend/sql/rls_policies.sql are reachable at all in practice.

**Stale webhook location (applies to web/'s pre-restructure code vs. current backend/).** .agents/decisions.md's DEC-004 states the Stripe webhook handler lives at "/api/webhooks/stripe" — a Next.js-style route, reflecting the old single-app layout.
