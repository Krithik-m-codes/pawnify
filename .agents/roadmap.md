# Pawnify Roadmap

Status as of 2026-07-11. This document is the master planning and technical-debt reference for the Pawnify monorepo. It is built from direct inspection of the current working tree; every factual claim below — file paths, field names, route paths, thresholds, library names, config values, line ranges — was re-verified directly against current source for this revision (including several commands actually run against the repo: a live `npm run build` / `node dist/main.js` in backend/, and `npx prisma generate` / `npx next build` in web/, described where relevant below). This pass corrected a small number of concrete errors found in the prior draft: two file-path misattributions (LICENSE and ARCHITECTURE.md are at the repository root, not under web/), two off-by-one line counts, an HTTP-route count that undercounted by two, one incorrect field-level claim about better-auth's additionalFields, an inaccurate framing of a repository-wide marker search, and an overreach in one claim about which env files define DATABASE_URL. Where a fact could not be verified from the repository, it is marked "Unknown" rather than guessed.

Subsystem convention used throughout: web/ refers to the mature Next.js 15-targeted (App Router, Prisma, Supabase, better-auth, Redux Toolkit Query) full-stack application under web/. backend/ refers to the new, partial NestJS API under backend/, which as of this writing implements exactly 8 modules on disk (backend/src/modules: auth, cron, customers, health, market-rates, organizations, storage, webhooks — verified by direct directory listing). Every claim below states which subsystem(s) it applies to. backend/ is not a complete API and should not be read as one; web/'s server-side logic is not deprecated merely because backend/ exists — in every domain examined, web/ still holds the actively-used implementation.

There is no monorepo tooling anywhere in the repository (no turbo.json, nx.json, pnpm-workspace.yaml, or lerna.json) and no CI/CD of any kind (no .github directory) — confirmed by direct filesystem inspection. backend/ and web/ are two fully independent npm projects, each with its own package-lock.json, joined only by convention and by the repo-root docker-compose.yml, which itself does not build successfully today (see INF-1).

Priority legend: P0 = critical, address before anything else in its area; P1 = high; P2 = medium; P3 = low. Difficulty legend: Easy/Medium/Hard, reflecting engineering effort, not urgency.

## 1. Subsystems at a Glance

backend/ (backend/src/modules/auth, cron, customers, health, market-rates, organizations, storage, webhooks) exposes 15 HTTP routes under a global /api prefix (backend/src/main.ts), plus an unguarded Swagger UI at /docs. Per-module route count: auth 2 (GET /auth/me, GET /auth/session), cron 2 (GET/POST /cron/update-rates), customers 1 (GET /customers/search), health 1 (GET /health), market-rates 1 (GET /market-rates), organizations 6 (GET/PUT :orgId/policy, GET :orgId/members, POST :orgId/members/invite, PATCH :orgId/members/:userId/role, DELETE :orgId/members/:userId), storage 1 (POST /storage/upload), webhooks 1 (POST /webhooks/dodo-payments) — 2+2+1+1+1+6+1+1 = 15, enumerated directly from every @Get/@Post/@Put/@Patch/@Delete decorator in backend/src/modules/**/*.controller.ts. It authenticates by first checking a bearer token or a cookie literally named "better-auth.session_token" against the shared session table (backend/src/modules/auth/guards/jwt-auth.guard.ts, backend/src/modules/auth/auth.service.ts, backend/src/modules/auth/auth.repository.ts), i.e. it deliberately trusts web/'s existing better-auth sessions rather than requiring a separate login system, and only falls back to verifying a Supabase-issued JWT if that lookup misses. It never verifies its own independently-signed JWT_SECRET token — that variable is declared in backend/.env.example but has zero references anywhere in backend/src (confirmed by repo-wide search), so it is dead configuration, not a live auth mechanism. This directly answers the question of how backend/ authenticates: it is a dual better-auth-session/Supabase-JWT verifier, not a self-signed-JWT verifier.

backend/'s own Prisma wiring currently prevents it from booting at all in the state checked into this repository: PrismaService (backend/src/database/prisma.service.ts) subclasses PrismaClient with no constructor override (an implicit empty super()), backend/prisma/schema.prisma's datasource block declares only provider = "postgresql" with no url and no env() reference, backend/package.json has no @prisma/adapter-pg dependency (it does depend on @prisma/client ^7.8.0 and pg ^8.22.0, but never wires the latter into the former), and there is no backend/prisma.config.ts — all confirmed by direct inspection for this document. Prisma Client 7.x requires either a static datasource url or an explicit adapter/datasourceUrl option; backend/ supplies neither, so `new PrismaClient()` throws PrismaClientInitializationError. This was reproduced directly for this document: `npm run build` (nest build) succeeds silently, and `node dist/main.js` logs NestFactory/InstanceLoader startup lines, then throws `PrismaClientInitializationError: \`PrismaClient\` needs to be constructed with a non-empty, valid \`PrismaClientOptions\`` from `new PrismaService` (dist/database/prisma.service.js:13, per the stack trace), and exits before any "listening on port" log line. Because DatabaseModule is @Global() and eagerly imported into AppModule's imports array (backend/src/app.module.ts), this blocks every route in every module. Contrast web/src/lib/db.ts, which constructs a pg.Pool from DATABASE_URL, wraps it with @prisma/adapter-pg's PrismaPg adapter, and passes `new PrismaClient({ adapter })` — the pattern backend/ would need to replicate.

web/ (web/src/app, web/src/lib, web/prisma) is the mature application and still owns essentially all business logic: loans, payments, followups, billing, settings, staff, reports, and dashboard. None of these eight feature areas has a backend/ equivalent beyond thin, partially-overlapping slices (see Section 3). web/'s own build is separately broken today for unrelated reasons: web/package.json pins "next": "^9.3.3" (matching the installed web/node_modules/next/package.json), a version that predates the App Router entirely, while web/src/app is a complete App Router tree and the same package.json's eslint-config-next devDependency is pinned to 16.2.10 — strong evidence the next pin is a regression, not an intentional downgrade (git history shows the pre-restructure root package.json pinned "next": "16.2.10" exactly, confirmed via `git show HEAD:package.json`). Reproduced directly for this document: `npx next build` in web/ fails immediately with `Error: Configuring Next.js via 'next.config.ts' is not supported. Please replace the file with 'next.config.js'.` — Next 9.3.3 predates TypeScript config-file support entirely, an independent symptom of the same version regression (rather than, or in addition to, an App-Router-support failure). Separately, web/prisma/schema.prisma's datasource block has the same missing-url problem as backend/'s copy — and unlike backend/, web/ does have its own prisma.config.ts declaring a datasource.url sourced from DIRECT_URL or DATABASE_URL, but this does not actually resolve the gap: reproduced directly for this document, `npx prisma generate` in web/ fails with `Error: Prisma schema validation - (get-config wasm) ... Argument "url" is missing in data source block "db"` (Prisma CLI 6.19.3) despite prisma.config.ts being loaded. web/next.config.ts also never sets output: "standalone", so web/Dockerfile's runner-stage copy of .next/standalone would fail even if the two build-blocking errors above it were fixed. All these failures were reproduced directly for this document; none of this is committed to git yet (git status shows web/, backend/, docker-compose.yml, .agents/, LICENSE, and CONTRIBUTING.md as untracked, with HEAD still reflecting the pre-restructure single-app layout — confirmed via `git status --short` and `git log --oneline`), so it cannot be assumed to represent whatever (if anything) is live in production today — that is Unknown from source alone.

Licensing: the repository-root LICENSE file (not under web/ or backend/ — confirmed by repo-wide search, only one LICENSE file exists) is Business Source License 1.1, Licensor "Pawnify Cloud Inc.", Change Date 2029-07-01 converting to MIT. As of today (2026-07-11) the BSL terms are still in force, with roughly three years remaining before the MIT conversion date. This is internally consistent and not itself a defect.

## 2. Implemented Features

### 2.1 web/ (mature Next.js application)

Multi-tenant schema and RLS SQL (schema level). web/prisma/schema.prisma models Organization, Branch, LoanPolicy, and DocumentType as a genuine multi-tenant layer, and backend/sql/rls_policies.sql enables Postgres Row-Level Security on 12 tenant tables (Organization, Branch, LoanPolicy, DocumentType, Customer, KycDocument, Loan, LoanItem, LoanCharge, Payment, LedgerEntry, FollowUp), gated by a current_user_organization_id() SQL function reading the app.current_organization_id session setting (with an auth.uid() fallback). Important caveat, not a separate claim: the one application-level helper that arms this session variable, runWithTenantContext (web/src/lib/supabase/server.ts, lines 32-44), has zero callers anywhere in the repository, and none of the audited read services (customers.ts, loans.ts, dashboard.ts) filter by organizationId either. So the RLS design exists at the schema level but is not connected to any live query path today — see AV-1 and MT-5, the two most severe findings in this document. Do not read "RLS is implemented" as "tenant isolation is enforced" — those are two different claims and only the first is currently true.

Decimal-only financial and weight math. web/src/lib/services/valuation.ts, interest.ts, and payments.ts perform every calculation through Prisma.Decimal, matching web/prisma/schema.prisma's Decimal(12,2) monetary columns and Decimal(10,4) weight columns. No floating-point money or weight math was found in these services.

Atomic payment waterfall. web/src/lib/services/payments.ts's recordPayment() allocates a payment to outstanding charges, then accrued interest, then principal, inside one transaction upgraded to Serializable isolation with retry-on-conflict (web/src/lib/db.ts's runSerializable, retrying on Prisma error P2034). It rejects overpayment beyond a 0.01 rounding tolerance rather than silently applying a partial payment, and writes an immutable LedgerEntry in the same transaction. Test coverage for this function itself is a separate, serious gap — see MT-1.

Loan lifecycle. web/src/lib/services/loans.ts implements createLoan (server-side item revaluation that never trusts client-submitted valuations, tiered LTV cap via getLtvSlabs/getLtvPercent, a mandatory-primary-ID gate via checkPanRequired, and a billing-plan quota gate via checkCanCreateLoan, all inside one Serializable transaction), closeLoan (requires zero outstanding principal, zero unsettled charges, and near-zero accrued interest), and releaseItems (a separate, later step for physical collateral release). deriveLoanDisplayStatus computes OVERDUE at read time from dueDate plus gracePeriodDays; the Prisma LoanStatus enum itself only has ACTIVE and CLOSED, which is a deliberate design choice per the code's own comments, not an oversight. Both loans.ts and this function are extensively untested (MT-2) and getLoans's ACTIVE/OVERDUE branch has a serious performance problem (PERF-2).

Interest accrual. web/src/lib/services/interest.ts computes simple daily interest from principalOutstanding and monthlyRate, accrued on read rather than persisted as a running balance. One documented drift: the file's own header comment and the Prisma DayCountConvention enum both list THIRTY_360 as a supported convention, but computeDailyInterest's actual branch only distinguishes ACTUAL_360 from a 365-divisor "everything else" — THIRTY_360 silently behaves identically to ACTUAL_365 today. This is a genuine spec-versus-code drift of exactly the kind this repository's own .agents/lessons.md warns is a recurring risk, though it is not severe enough to have been scored as its own register row here.

Authentication. web/src/lib/auth.ts constructs betterAuth() with a Prisma adapter; web/prisma/schema.prisma contains the user/session/account/verification tables better-auth expects. Of the three custom fields declared in auth.ts's user.additionalFields, role and isActive are marked input: false so they cannot be client-set at signup (the code comment above them says exactly this: "role/isActive must never be settable from client sign-up payloads"); phone is marked input: true and can be client-set. This is the mechanism backend/ piggybacks on (Section 1).

Jurisdiction and worldwide-policy configuration. web/src/app/(app)/onboarding/onboarding-wizard-client.tsx's PRESETS array offers 9 real jurisdiction presets plus Custom (US, GB, EU, AE, CH, SG, AU, CA, India), persisted per-organization into LoanPolicy via saveLoanPolicyAction (web/src/app/(app)/onboarding/actions.ts). A second, only partially overlapping registry, WORLDWIDE_JURISDICTIONS (web/src/lib/config/jurisdictions.ts, 12 entries: US, GB, EU, AE, CH, SG, AU, CA, IN, SA, JP, GLOBAL), is consumed by the onboarding wizard and by universal-asset-calculator.tsx (the public marketing calculator), but is not the same source of truth as PRESETS. Important gap: LoanPolicy's weightUnit, purityExpression, and dayCountConvention fields are captured at onboarding but are not consistently read back — createLoan always calls computeItemValuation with the GRAM default (no weightUnit field exists on LoanItemInput at all), normalizePurityPercent has no call site anywhere in web/src, and no loans.ts/payments.ts call site ever supplies a dayCountConvention override to the interest functions (in fact the LoanForInterest objects they build never even set a dayCountConvention field, so computeAccruedInterest/computeInterestSummary always fall through to their own ACTUAL_365 default). The "worldwide rules engine" is real in the data model and at onboarding time, but is not fully enforced by the services that would need to consult it afterward. Separately, India-specific defaults leak into the supposedly worldwide runtime path: the Rupee symbol (₹) is hardcoded into payments.ts's overpayment error and ledger description, and utils.ts's formatDate/formatDateTime hardcode the en-IN locale with no override parameter.

Universal asset categories (marketing scaffolding). web/src/lib/config/asset-categories.ts defines 6 categories (Precious Metals & Bullion, Horology & Luxury Watches, Fine Art & Collectibles, Vehicles & Heavy Equipment, Luxury Designer Goods & Electronics, General Appraised Collateral) with example items and default LTV percentages (80/65/50/55/60/50) that do not match the 85/80/75 percent tiers actually used by valuation.ts. It is consumed only by a marketing calculator and its own test — the real loan-creation path is hardcoded to MetalType GOLD/SILVER only (web/prisma/schema.prisma's MetalType enum; web/src/lib/validation/loan.ts's `z.enum(["GOLD", "SILVER"])`).

Billing/plan quotas. web/src/lib/services/billing.ts defines PLAN_QUOTAS for four plans (self_hosted/starter/growth/enterprise) with per-plan maxActiveLoans/maxBranches. checkCanCreateLoan is a real, wired-in gate inside createLoan; checkCanCreateBranch exists with identical logic but has zero callers anywhere in web/src (dead code, noted under MT-8 rather than the dead-code register since it is bundled with a missing-tests finding). updateOrganizationPlan is called from a platform-admin action, not from backend/'s Dodo Payments webhook (formerly Stripe; see Section 6), which has its own, entirely separate implementation of "write Organization.billingPlan" (backend/src/modules/webhooks/webhooks.repository.ts) — two independent code paths perform the same mutation.

Dashboard, reports, followups, staff, settings, customers/KYC. All of these are real, working features implemented entirely in web/ (web/src/lib/services/dashboard.ts; web/src/app/(app)/reports/actions.ts; web/src/app/(app)/followups/actions.ts; web/src/app/(app)/admin/staff/actions.ts; web/src/app/(app)/admin/settings/actions.ts plus web/src/lib/services/settings.ts; web/src/lib/services/customers.ts plus web/src/lib/validation/customer.ts). Every one of them has confirmed, non-trivial defects cataloged in the register (unscoped cross-tenant queries, unbounded table scans, business logic embedded directly in Server Actions instead of a service layer, missing input validation, or zero test coverage) — see Section 3 and the register rows cross-referenced there. Customer/KYC validation is hardcoded to Indian document formats (Aadhaar, PAN, Voter ID, Passport, Driving Licence regexes in web/src/lib/validation/customer.ts), and the Prisma KycDocType enum's sixth value, CUSTOM, cannot pass that schema's `docType` zod enum as written (which lists only AADHAAR/PAN/VOTER_ID/PASSPORT/DRIVING_LICENSE).

Test suite. web/src/__tests__ contains seven files (billing, interest, jurisdiction, rls-isolation, universal-assets, valuation, waterfall.test.ts — confirmed by directory listing), more than either web/README.md or the repository-root ARCHITECTURE.md claims (ARCHITECTURE.md lives at the repo root, not under web/, and names only three of the seven — valuation.test.ts, interest.test.ts, waterfall.test.ts — while web/README.md names none by filename, only describing three conceptual areas). Coverage is real for the pure math (valuation, interest, jurisdiction) but two of the seven are misleading: waterfall.test.ts never imports the real recordPayment/previewPaymentAllocation from payments.ts (it re-declares its own local computePaymentWaterfall helper), and rls-isolation.test.ts filters a hardcoded in-memory array declared inline in the test and never imports anything but vitest itself — see MT-1 and MT-5.

Marketing site. A public landing page, pricing page, open-source page, and docs shell exist and reference the jurisdiction/asset-category registries above. Two marketing pages make strong RLS-enforcement claims that the AV-1/MT-5 findings directly contradict as currently wired: web/src/components/marketing/features-section.tsx states, verbatim, "Even application-layer bugs cannot leak cross-tenant loan or customer records" (under a "Load-Bearing Row-Level Security" heading, next to a separate "RLS Verified on Every Query" badge), and web/src/app/open-source/page.tsx separately claims "Tenant isolation lives inside the PostgreSQL engine via session variables (`app.current_organization_id`), protecting data at the deepest layer."

### 2.2 backend/ (partial NestJS API)

Eight modules exist on disk (backend/src/modules/auth, cron, customers, health, market-rates, organizations, storage, webhooks), exposing 15 routes total under the /api prefix (2 auth + 2 cron + 1 customers + 1 health + 1 market-rates + 6 organizations + 1 storage + 1 webhooks), plus Swagger documentation at /docs (backend/src/main.ts, unguarded). Per module:

auth — no dedicated business feature; provides GET /auth/me and GET /auth/session (backend/src/modules/auth/auth.controller.ts) plus the JwtAuthGuard/RolesGuard/AuthService/AuthRepository stack every other guarded controller depends on. Its Supabase-JWT fallback path (auth.repository.ts's verifySupabaseJwtUser) fabricates a role of "STAFF" and isActive: true for a Supabase-authenticated user with no matching local user row, with no further server-side check. Zero test coverage (MT-6).

cron — GET/POST /cron/update-rates behind CronAuthGuard, which fails open (allows all requests) when CRON_SECRET is unset (SEC-12); delegates to market-rates' fetch logic. web/vercel.json's only cron entry still points at a stale path (/api/cron/update-rates, which resolves to web/'s own, now-removed API route — web/src/app/api contains only auth/[...all]/route.ts) rather than at this backend/ endpoint, so nothing in the repository currently schedules this route automatically. Zero test coverage (MT-11).

customers — a single GET /customers/search typeahead endpoint (backend/src/modules/customers/customers.controller.ts), correctly gated by JwtAuthGuard but with no organizationId filter in its Prisma query, a real cross-tenant leak (folded into AV-1/SEC-4's broader RLS-gap framing rather than scored as its own row). Full customer CRUD, KYC, and PAN-threshold logic remain entirely in web/. Zero test coverage (MT-15).

health — public GET /health running SELECT 1 via Prisma; leaks raw driver error text on failure (SEC-10). Zero test coverage (MT-18).

market-rates — public GET /market-rates plus a fetchAndSaveLiveMetalRates() method invoked only from cron; independently reimplements the identical fallback chain already in web/src/lib/services/market-rates.ts, with confirmed behavioral divergences (D-1, AV-12). Zero test coverage (MT-10).

organizations — 6 RBAC-intended routes for LoanPolicy and team-member management. Its guard chain is broken (SEC-1: the class applies only RolesGuard, never JwtAuthGuard, so the two GET routes are fully unauthenticated and the four @Roles-guarded write routes always throw 401), it performs no tenant-membership check against the orgId in the URL, and its invite-by-email logic can silently reassign an existing user from a different organization (SEC-6). Its web/ RTK Query consumer, organizationsApi.ts, has zero callers in the actual UI (DC-1). Zero test coverage (MT-14).

storage — POST /storage/upload, correctly gated by JwtAuthGuard, but with no file-size limit, no mimetype/extension allow-list, and a free-text bucket parameter with no allow-list (SEC-2, MV-9); no repository layer of its own, unlike every sibling module (AV-7). Zero test coverage (MT-16).

webhooks — POST /webhooks/dodo-payments (migrated from Stripe), with no JwtAuthGuard, as expected for a public third-party webhook receiver, but now verifying a real HMAC-SHA256 signature via the `dodopayments` SDK's `unwrap()` before trusting the payload (MV-1, RESOLVED — see Section 8.9); the old request DTO with zero class-validator decorators no longer exists, since the endpoint now binds the raw body instead of a `@Body()` DTO. Zero test coverage (MT-7).

A second, currently byte-identical copy of web/'s Prisma schema and seed script lives at backend/prisma/schema.prisma and backend/prisma/seed.ts, hand-maintained in parallel with no shared package, symlink, or codegen step (D-2, AV-3).

## 3. Partially Implemented Features: the Core Migration Gap

This is the central fact about where this migration currently stands: backend/ has not absorbed any of web/'s eight core feature areas beyond thin, partially-overlapping slices, and two of those slices (organizations, market-rates) are themselves broken or behaviorally divergent from web/'s originals.

|---|---|---|---|
| Loans | web/src/lib/services/loans.ts (525 lines): creation, closure, item release, LTV/valuation enforcement, PAN gate, billing quota gate | None. No loans module, no Loan-model Prisma reference anywhere in backend/src (confirmed by repo-wide search for prisma.loan/loanItem/loanCharge calls; the only textual hits for the word "Loan" are two Swagger description strings in organizations' policy DTO) | Entire domain unmigrated. backend/'s organizations module only manages LoanPolicy (per-org configuration), never a Loan record itself. |
| Payments | web/src/lib/services/payments.ts (284 lines): atomic charges-then-interest-then-principal waterfall, receipt numbering, ledger writes | None | Entire domain unmigrated. |
| Followups | web/src/app/(app)/followups/actions.ts (151 lines): reminder/follow-up task CRUD | None | Entire domain unmigrated. |
| Billing | web/src/lib/services/billing.ts (138 lines) plus a platform-admin action for plan changes | backend/'s webhooks module writes Organization.billingPlan from a Dodo Payments payload that is now verified via a real signature check before being trusted (backend/src/modules/webhooks/webhooks.repository.ts; see MV-1, RESOLVED) | Two independent, non-communicating code paths mutate the same field; neither calls the other. Reading this table alongside Section 2.2: backend/'s useful net-new surface area today is authentication interoperability, Swagger documentation, a narrow customer-search endpoint, and a cron-triggered market-rate refresh — not a parallel implementation of Pawnify's lending business. Anyone extending backend/ should assume web/'s service-layer code (web/src/lib/services/*.ts) is the actively-used implementation for all eight areas above unless and until backend/ demonstrably replaces it end to end.

## 4. TODOs and FIXMEs Found in Code

A repository-wide search for TODO, FIXME, XXX, HACK, and @deprecated markers across backend/src and web/src (all .ts/.tsx files) returns exactly one match in the entire scope: the string "XXXXX" inside a comment in web/src/lib/services/payments.ts describing the receipt-number format ("REC-YYYYMMDD-XXXXX") — a coincidental substring match on the "XXX" pattern, not a genuine marker of unfinished work. backend/src alone has zero matches for any of the five patterns.

Two other facts are worth recording separately, but neither is actually a hit from this literal search (neither string contains "TODO", "FIXME", "XXX", "HACK", or "@deprecated"): HttpStatus.NOT_IMPLEMENTED, a real NestJS enum member name used in backend/src/modules/webhooks/webhooks.service.ts to return 501 when DODO_PAYMENTS_API_KEY or DODO_PAYMENTS_WEBHOOK_KEY is unset (already covered by MV-1, RESOLVED); and a section-label comment, "Legal & Compliance Stubs", above a working set of links to /privacy and /terms in web/src/components/marketing/footer.tsx (both pages resolve to real page.tsx files; "stubs" describes their content maturity, not missing functionality, and is out of scope for this technical audit).

This codebase does not use conventional in-line comment markers to flag unfinished work. That does not mean the codebase is complete — the 85-item register in Section 8 documents extensive unfinished and drifted functionality — it means none of it is signposted in the source itself. A future contributor grepping for TODO specifically will find nothing at all; the Detected Issues Register in this document is currently the only place these gaps are tracked.

## 6. Historical .agents/ Notes vs Current Reality

The five files at .agents/context.md, .agents/decisions.md, .agents/lessons.md, .agents/schema.md, and .agents/tasks.md describe a pre-restructure, single-app version of Pawnify, before backend/ existed, and never mention backend/ at all. Per this document's own rules, current code wins wherever they conflict; the conflicts below are the ones worth recording explicitly rather than silently overwriting:

- .agents/tasks.md marks every item in Phase 0 through Phase 6 as done ([x]), including "Implement load-bearing Postgres Row-Level Security (RLS) policies for tenant isolation" (Phase 1) and "RLS cross-tenant isolation verification (src/__tests__/rls-isolation.test.ts)" (Phase 5) — both exact quotes from that file. Current code shows the RLS session variable is never armed by any live query path and the cited test never touches a database (AV-1, MT-5). The schema-level RLS work is real; the enforcement and verification claims are not currently true.
- .agents/decisions.md's DEC-004 states quota enforcement plus "a Stripe webhook handler (/api/webhooks/stripe)" is "RESOLVED & EXECUTED" — both exact quotes, describing the pre-backend/-split, single-app Stripe integration. The quota-enforcement half (web/src/lib/services/billing.ts) is real.
