### DEC-001: Row-Level Security (RLS) Enforcement & Session Variables
- **Status**: RESOLVED & EXECUTED (Phase 1, pre-backend/ split)
- **Decision**: PostgreSQL Row-Level Security is load-bearing (originally recorded at sql/rls_policies.sql). Multi-tenancy is enforced inside PostgreSQL by setting a session context variable (`app.current_organization_id`) inside transactions and queries.
- **Verification (original)**: Tested via src/__tests__/rls-isolation.test.ts (path as originally recorded, pre-restructure flat layout).
- **Current-code status (2026-07-11)**: Still true and load-bearing in web/. The RLS SQL file has physically relocated to backend/sql/rls_policies.sql even though the session-variable-setting side of the mechanism is only exercised from web/'s code path — a split this decision did not anticipate. web/src/lib/supabase/server.ts's `runWithTenantContext()` sets the session variable via `SELECT set_config('app.current_organization_id', $1, true)` inside a Prisma transaction (confirmed verbatim); backend/sql/rls_policies.sql enables `ROW LEVEL SECURITY` on exactly 12 tenant tables (Organization, Branch, LoanPolicy, DocumentType, Customer, KycDocument, Loan, LoanItem, LoanCharge, Payment, LedgerEntry, FollowUp — counted directly), gated by a `current_user_organization_id()` helper function that reads that same setting (falling back to a lookup on the `user` table by `auth.uid()` if the setting is unset). The original test file is now at web/src/__tests__/rls-isolation.test.ts. **Caveat**: a repository-wide search of backend/src for `current_organization_id` and `set_config` returns zero matches — nothing in backend/ ever sets this session variable (see DEC-006's "known risk" note). RLS enforcement is currently only exercised through web/'s code path, and whether it also protects backend/'s queries depends on which Postgres role backend/'s `DATABASE_URL` connects as, which is Unknown from static source.

### DEC-003: Self-Hosted & Cloud Database Support Paths
- **Status**: RESOLVED & EXECUTED (Phase 4, pre-backend/ split)
- **Decision**: Pawnify supports two deployment topologies: (1) complete self-hosted Docker Compose orchestration (docker-compose.yml) running local PostgreSQL 16 plus the Next.js app; (2) Bring-Your-Own (BYO) PostgreSQL / Supabase Cloud instance configured via .env.
- **Current-code status (2026-07-11)**: Topology (2) is real and current for backend/: backend/.env.example directly defines both `DATABASE_URL` and `SUPABASE_URL` against an externally-provisioned instance. It is only partially documented for web/: none of web/'s three checked-in env files (web/.env, web/.env.example, web/.env.prod — all three read directly) define `DATABASE_URL` or `DIRECT_URL`, the variables both web/prisma.config.ts and web/src/lib/db.ts read from `process.env` at the CLI and runtime layers respectively. web/.env.example instead documents `NEXT_PUBLIC_SUPABASE_URL`/`NEXT_PUBLIC_SUPABASE_ANON_KEY` (a browser-side Supabase client, used only for Supabase Auth/Storage — see DEC-008), `BETTER_AUTH_SECRET`/`BETTER_AUTH_URL`, and `NEXT_PUBLIC_API_URL`/`NEXT_PUBLIC_BACKEND_URL` (pointing at backend/). A self-hoster following web/.env.example literally would have no documented source for the connection string web/'s own Prisma/pg pool requires at runtime; whether it is meant to be supplied by some other, untracked mechanism is Unknown from static source. Topology (1) is present as a file but does not work end-to-end today: docker-compose.yml's `app` service builds from a root-level Dockerfile that does not exist anywhere in the repository (the only Dockerfile is web/Dockerfile, confirmed by repository-wide search), and its `db` service's RLS-policy volume mount (`./sql/rls_policies.sql`) targets a root sql/ directory that does not exist (the real file is backend/sql/rls_policies.sql). docker-compose.yml also defines no service at all for backend/, a service this decision predates. Independently of Docker, web/'s own build is currently broken on its own terms: web/package.json pins `next` to `^9.3.3` (a pre-App-Router version, confirmed installed in web/node_modules at exactly 9.3.3) while web/src/app is a complete App Router tree — and, as a third, independently-confirmed data point on the same confusion, that same package.json pins the devDependency `eslint-config-next` to `16.2.10`, so the file's own three signals (runtime dependency, installed package, lint-config version) do not agree on what Next.js major version this app targets. Separately, web/prisma/schema.prisma's datasource block declares no `url` at all (just `provider = "postgresql"`), which web/prisma.config.ts attempts to inject via `process.env["DIRECT_URL"] || process.env["DATABASE_URL"]` — but per above, neither variable is defined in any of web/'s checked-in env files, so as committed, this mechanism has nothing to read. This decision's intent is unchanged, but neither of its two implementations currently succeeds as committed to the working tree — and per the provenance note above, none of this has ever been committed at all.

### DEC-004: SaaS Billing & Plan Quotas
- **Status**: RESOLVED & EXECUTED (Phase 4, pre-backend/ split)
- **Decision**: Implemented server-side quota enforcement (src/lib/services/billing.ts) with a Stripe webhook handler (/api/webhooks/stripe).
  - Self-Hosted OSS Mode (`billingPlan === null`): unlimited active loans and branches ($0 forever).
  - Cloud Managed Tiers: Starter (100 loans / 1 branch), Growth (1,000 loans / 5 branches), Enterprise (unlimited).
- **Current-code status (2026-07-11)**: Quota enforcement is confirmed still real, now at web/src/lib/services/billing.ts. `PLAN_QUOTAS` defines `self_hosted` (Infinity loans / Infinity branches / $0), `starter` (100 / 1 / $49 per month), `growth` (1,000 / 5 / $149 per month), and `enterprise` (Infinity / Infinity / $399 per month) — all confirmed verbatim; `checkCanCreateLoan` / `checkCanCreateBranch` enforce these against `prisma.loan.count` / `prisma.branch.count` scoped by `organizationId`. **The webhook handler has moved subsystems, and its provider has since changed too**: no /api/webhooks/stripe route exists under web/src/app/api today (the only route there is the better-auth catch-all, web/src/app/api/auth/[...all]/route.ts — confirmed to be the only file anywhere under that directory). The live handler is now backend/src/modules/webhooks/webhooks.controller.ts's `POST /webhooks/dodo-payments` (the integration was migrated off Stripe onto Dodo Payments in a later pass), calling webhooks.service.ts to update `Organization.billingPlan` via WebhooksRepository on `subscription.active`/`subscription.renewed` (upgrade) and `subscription.cancelled`/`subscription.expired`/`subscription.failed` (clear) events — a split (and, later, a provider swap) this original decision did not anticipate (quota read-path stays in web/; billing-plan write-path moved to backend/, both against the same `organizationId`-scoped column). **RESOLVED (this pass) — previously recorded here as a confirmed gap**: webhooks.service.ts now constructs the official `dodopayments` SDK client and calls `client.webhooks.unwrap(rawBody.toString(), { headers })` — a genuine HMAC-SHA256 signature check per the Standard Webhooks spec — before trusting `payload.type`/`payload.data.metadata`; the endpoint binds `@Req() req: RawBodyRequest<Request>` rather than a validated DTO, with raw-body capture enabled via `NestFactory.create(AppModule, { rawBody: true })` in backend/src/main.ts (needed because signature verification requires the exact original bytes, not Nest's re-serialized JSON). `unwrap()` throws on an invalid/missing signature, which the service turns into a real `401 Invalid webhook signature` response — verified directly: a live curl POST with no valid signature headers against the running dev server returned exactly that 401. The endpoint still returns HTTP 501 if `DODO_PAYMENTS_API_KEY`/`DODO_PAYMENTS_WEBHOOK_KEY` is unset (the same fail-closed shape as the old Stripe-secret check, just new variable names). A caller can no longer set an organization's `billingPlan` by simply POSTing a JSON body shaped like the old (now-deleted) `StripeWebhookPayloadDto` — the endpoint no longer binds the body via `@Body()`/a DTO at all, and an invalid-signature request is rejected before any payload field is ever read.

### DEC-006: Extraction of a Standalone NestJS API Service (backend/)
- **Status**: INFERRED — no explicit design record found; reconstructed from code evidence only.
- **Decision as observed**: A second, independent application, backend/, was created as a NestJS 11 API with its own package.json, tsconfig, and ESLint config, exposing 8 feature modules (backend/src/modules: auth, cron, customers, health, market-rates, organizations, storage, webhooks — confirmed both via directory listing and backend/src/app.module.ts's imports). It runs as a separate process on its own port (3001 by default, per backend/.env.example and backend/src/main.ts's fallback) alongside web/ (port 3000), rather than as new routes inside web/'s own App Router — web/src/app/api today contains only the better-auth catch-all, web/src/app/api/auth/[...all]/route.ts.
- **Evidence this is a genuine, if partial, migration rather than a parallel prototype**:
  - backend/'s auth layer is deliberately built to interoperate with web/'s existing better-auth login system, not replace it. Directly confirmed by reading all three files: backend/src/modules/auth/guards/jwt-auth.guard.ts extracts a bearer token or, failing that, the literal `better-auth.session_token` cookie; backend/src/modules/auth/auth.service.ts's `validateToken()` checks that token against the shared `session` table first (backend/src/modules/auth/auth.repository.ts's `findSessionByToken`, raw SQL over a plain node-postgres `Pool` wrapped inside SupabaseService's own `query()` helper), and only falls back to Supabase JWT verification (`verifySupabaseJwtUser`) if that lookup misses. There is no independent backend/-only login system. (One caveat: web/src/lib/auth.ts, which configures better-auth, never itself states the cookie name `better-auth.session_token` — that file has no explicit cookie configuration at all, so the name is presumed to be better-auth's own library default rather than something visible in web/'s source; this repository does not surface that default directly since node_modules is out of scope for this review.)
  - At least one backend/ endpoint is already live-wired into a real web/ UI flow, not merely scaffolded: web/src/app/(app)/loans/new/page.tsx calls `useLazySearchCustomersQuery` (web/src/lib/redux/api/customersApi.ts), which issues `GET /customers/search` against backend/ via web/src/lib/axiosClient.ts (base URL defaulting to `http://localhost:3001/api`) — confirmed by direct reading of all three files. Tellingly, customersApi.ts mixes both migration patterns side by side in the same file: `searchCustomers` is the only endpoint wired to a real `url` (routed through axiosClient to backend/), while every other endpoint in that same file (getCustomerById, getCustomers, createCustomer, updateCustomerDetails, deleteCustomer, addKycDocument, verifyKycDocument) still uses an `{ action: <serverActionFn>, args }` shape that calls web/'s own pre-existing Server Actions directly, in-process. Both shapes are handled by the same generic "hybrid" base query (web/src/lib/redux/api/baseApi.ts's `hybridAxiosBaseQuery`, confirmed: it branches on whether an endpoint's `query()` returns a `url` or an `action`) — this hybrid dispatcher is the literal, reusable plumbing that lets individual RTK Query endpoints be cut over to backend/ one at a time without changing any calling component. This is running evidence of an incremental, "strangler fig"-style extraction already in effect, not a hypothetical one.
  - A second slice — organization policy and team-member management — has been built server-side (backend/src/modules/organizations/organizations.controller.ts's `GET`/`PUT :orgId/policy` and full member invite/list/role-update/remove routes) with a matching client module already written (web/src/lib/redux/api/organizationsApi.ts), but that module's hooks (`useGetOrganizationPolicyQuery`, `useGetTeamMembersQuery`, `useInviteTeamMemberMutation`, etc.) — and indeed the string "organizationsApi" itself — appear nowhere else in web/src (confirmed by repository-wide search: the only match is the file's own definition) — the equivalent UI (web/src/app/(app)/admin/settings, web/src/app/(app)/admin/staff) still calls its own pre-existing Server Actions, wrapped through unrelated RTK Query modules (settingsApi.ts, staffApi.ts, both using the same `{ action, args }` pattern described above) and has not been cut over. This slice is scaffolded on both ends but not yet connected — which, per the Known risk below, is arguably just as well: as currently coded, this controller could not safely be wired up yet.
  - backend/ also hosts a billing webhook receiver (backend/src/modules/webhooks/webhooks.controller.ts — originally a Stripe integration, since replaced end-to-end with Dodo Payments; see DEC-004's current-code status) and a cron-triggered market-rate refresh (backend/src/modules/cron/cron.controller.ts), both triggered by external callers (the payments provider, a scheduler) rather than a logged-in browser session — a plausible, but still inferred, reason to place this logic outside the Next.js request/response cycle. (One directly-verified aside: cron.controller.ts's `CronAuthGuard` fails *open* — `canActivate()` returns `true` unconditionally whenever `CRON_SECRET` is unset, rather than rejecting the request — the mirror image of webhooks.service.ts's fail-*closed* behavior when `DODO_PAYMENTS_API_KEY`/`DODO_PAYMENTS_WEBHOOK_KEY` is unset.)
- **Inferred rationale** (not stated anywhere in the repository): the most likely motivations for splitting a second API out of an already-capable full-stack Next.js app are (a) a request surface decoupled from Next.js's own render/Server-Action lifecycle for externally-triggered integrations (webhooks, cron), (b) a single documented, versioned HTTP contract (see DEC-007) other future clients could call without depending on Next.js internals, and (c) a place to consolidate cross-tenant/admin operations behind explicit guard/decorator-based authorization rather than checks scattered across Server Actions. All three are standard, plausible motivations; none is confirmed.
- **Known risk introduced by the current partial state** (directly verified, not inferred): two separate, independently-confirmed gaps, both on controllers cited as evidence above.
  1. The one endpoint DEC-006 confirms is already live in a production UI flow — backend/src/modules/customers/customers.repository.ts's `searchCustomers()` — runs `this.prisma.customer.findMany` with **no `organizationId` filter of any kind**, and its controller (backend/src/modules/customers/customers.controller.ts) is guarded only by `JwtAuthGuard`, an authentication check, not a tenant-scoping one. DEC-001's RLS was designed to backstop exactly this class of missing-WHERE-clause bug at the database layer, but nothing under backend/src ever sets the `app.current_organization_id` session variable DEC-001's policies key off. Whether Postgres RLS still silently protects this query depends entirely on which role backend/'s `DATABASE_URL` connects as — Unknown from static source, not executed to confirm.
  2. backend/src/modules/organizations/organizations.controller.ts — the scaffolded-but-not-yet-wired slice described above — carries **no `JwtAuthGuard` anywhere**, only `@UseGuards(RolesGuard)` at the class level. This is confirmed to be the *only* exception among backend/'s 8 controllers: a repository-wide search shows auth.controller.ts, customers.controller.ts, and storage.controller.ts all pair `JwtAuthGuard` with their route-level `@UseGuards`; organizations.controller.ts alone does not. Concretely, per roles.guard.ts's own logic: (a) `GET :orgId/policy` and `GET :orgId/members` carry no `@Roles()` decorator, so `RolesGuard.canActivate()` returns `true` immediately without ever inspecting `request.user` — these two routes are fully unauthenticated for any `orgId`, guessed or enumerated; (b) the four mutating routes (`PUT :orgId/policy`, `POST :orgId/members/invite`, `PATCH :orgId/members/:userId/role`, `DELETE :orgId/members/:userId`) do carry `@Roles('OWNER', 'ADMIN')`, so `RolesGuard` proceeds to read `request.user` — but nothing upstream of `RolesGuard` on this controller ever populates `request.user` (that is `JwtAuthGuard`'s job elsewhere in the codebase, and `JwtAuthGuard` never runs on this controller), so `request.user` is always `undefined` and `RolesGuard`'s own code throws `HttpException(401)` unconditionally. Net effect, confirmed by direct reading, not inferred: the two read routes leak an organization's financial policy and team-member PII (name, email, phone) to any caller who knows or guesses an `orgId`, with zero authentication; the four write routes are permanently unusable (always 401) regardless of caller identity or actual role. This is consistent with, and plausibly explains, why organizationsApi.ts's hooks are never imported anywhere in web/src (see above) — the feature could not have been exercised end-to-end from a real UI without hitting this. See DEC-007 for the related, corrected claim about which controllers `JwtAuthGuard` actually gates.

### DEC-007: NestJS as the Framework for backend/
- **Status**: INFERRED
- **Decision as observed**: backend/ is built on NestJS 11 (`@nestjs/common`, `@nestjs/core`, `@nestjs/platform-express` at `^11.0.1`, per backend/package.json) rather than additional Next.js Route Handlers inside web/, a lighter framework, or a different structured one.
- **Evidence of what NestJS specifically supplies that is actually in active use**:
  - Decorator-driven, DI-managed cross-cutting concerns: one global `ValidationPipe` (`whitelist`/`transform`/`forbidNonWhitelisted`, backend/src/main.ts) enforced against real class-validator decorators on DTOs (e.g. backend/src/modules/customers/dto/customer-search-query.dto.ts's `@IsOptional()`/`@IsString()`); one global exception filter normalizing every error into a single JSON envelope (backend/src/common/filters/http-exception.filter.ts's `AllExceptionsFilter`, `@Catch()` with no argument); one global logging interceptor (backend/src/common/interceptors/logging.interceptor.ts). Note these last two are wired via `app.useGlobalFilters(new AllExceptionsFilter())` / `app.useGlobalInterceptors(new LoggingInterceptor())` using plain `new` in backend/src/main.ts, not Nest's `APP_FILTER`/`APP_INTERCEPTOR` DI-token pattern (confirmed absent by repository-wide search) — a partial, not full, use of Nest's own idiom.
  - Guard-based authorization as a reusable, declarative primitive: `JwtAuthGuard` (backend/src/modules/auth/guards/jwt-auth.guard.ts) correctly gates auth.controller.ts's two routes and the entirety of customers.controller.ts and storage.controller.ts (all confirmed via `@UseGuards(JwtAuthGuard)`, each paired with `@ApiBearerAuth()`). A custom `@Roles(...)` decorator and `RolesGuard` reading it via `Reflector` (backend/src/modules/auth/decorators/roles.decorator.ts, backend/src/modules/auth/guards/roles.guard.ts) are applied as a one-line decorator per route on organizations.controller.ts's four mutating routes, evidently intended to gate them to `OWNER`/`ADMIN` only — confirmed directly by reading this controller. **Correction to an earlier draft of this document**: this pattern does not actually function on organizations.controller.ts as written, because `JwtAuthGuard` is never applied there (only `RolesGuard` is, at the class level) — see DEC-006's Known risk #2 for the full, directly-verified mechanics (unauthenticated reads, permanently-401 writes). The guard/decorator pattern is real, working NestJS machinery — it is simply misapplied on this one controller. web/'s equivalent authorization checks (e.g. web/src/lib/auth/session.ts's `checkAdmin()`) are plain functions called manually inside each Server Action; there is no first-class declarative-guard equivalent in web/.
  - Built-in, decorator-driven OpenAPI documentation: backend/src/main.ts wires `@nestjs/swagger`'s `DocumentBuilder` and mounts a live Swagger UI at `GET /docs`. `@ApiTags`/`@ApiOperation`/`@ApiResponse` decorators are present on all 8 controllers (confirmed by direct reading of each); `@ApiBearerAuth` is present on exactly 4 of the 8 (auth, customers, organizations, storage), correctly omitted from the other 4 (health and market-rates, which are unauthenticated by design; cron, which uses a shared-secret header via `CronAuthGuard` rather than a bearer JWT; and webhooks, intentionally public for the Dodo Payments callback) — a more precise claim than "present on every controller." No Swagger CLI plugin is registered in backend/nest-cli.json, so every documented field is hand-authored, not type-inferred — a real, functioning capability web/'s Server Actions have no equivalent of.
- **Inferred rationale**: NestJS's opinionated module/controller/service/DI structure, combined with its guard/interceptor/pipe/filter pipeline and first-class Swagger integration, is a standard choice when a team wants a conventionally-structured, independently-documented HTTP API with declarative authorization — exactly the pieces observed above. This is inferred from fit-to-what-was-built, not from a stated comparison to alternatives (Express, Fastify, tRPC, GraphQL); no such comparison appears anywhere in the repository.
- **Caveat on incompleteness of adoption**: several Nest idioms are only partially applied — no `APP_GUARD`/`APP_FILTER`/`APP_INTERCEPTOR` global-provider registration (confirmed absent by repository-wide search), no environment-variable validation schema (no Joi/class-validator `validate` option passed to `ConfigModule.forRoot()`, backend/src/config/config.module.ts), and most feature-specific env vars (`DODO_PAYMENTS_API_KEY`, `DODO_PAYMENTS_WEBHOOK_KEY`, `DODO_PAYMENTS_ENVIRONMENT`, `CRON_SECRET`, and all of market-rates' rate/LTV constants) bypass `ConfigService` entirely via direct `process.env` reads (backend/src/modules/webhooks/webhooks.service.ts, backend/src/modules/cron/guards/cron-auth.guard.ts, backend/src/modules/market-rates/market-rates.service.ts). This suggests backend/ is an early-stage scaffold still converging on its own conventions, not yet a mature reference implementation of Nest best practice — the organizations.controller.ts guard gap above is a symptom of the same immaturity.

### DEC-008: Continued Dual Use of Prisma and Supabase
- **Status**: Mechanism VERIFIED from current code; design intent INFERRED — no rationale recorded anywhere in the repository.
- **Decision as observed**: Both subsystems keep Prisma and the Supabase JS client/SDK as dependencies, and both are exercised at runtime — but the split is not "Prisma for migrations, Supabase for tenant CRUD" as the pre-restructure notes describe (see conflict note below). It is closer to: Prisma is the general-purpose typed ORM for domain queries; the Supabase client is reserved narrowly for the two capabilities Prisma itself does not provide — Supabase Auth's JWT/session verification, and Supabase Storage.
- **Evidence, backend/**: `PrismaService` (backend/src/database/prisma.service.ts, a bare `extends PrismaClient` with zero constructor arguments and no custom datasource wiring) is the live query engine for 5 of backend/'s 8 modules — health (backend/src/modules/health/health.repository.ts's raw `SELECT 1` via `$queryRaw`), customers (backend/src/modules/customers/customers.repository.ts), market-rates (backend/src/modules/market-rates/market-rates.repository.ts), organizations (backend/src/modules/organizations/organizations.repository.ts), and webhooks (backend/src/modules/webhooks/webhooks.repository.ts) — all confirmed by direct reading and by a repository-wide search showing exactly these 5 files import `PrismaService`. `SupabaseService` (backend/src/database/supabase.service.ts) is used by exactly two modules — confirmed by the same kind of search — backend/src/modules/auth/auth.repository.ts (raw parameterized SQL against the better-auth `session`/`user` tables, executed through a plain node-postgres `Pool` that `SupabaseService.query()` wraps internally, plus `supabase.auth.getUser(token)` for JWT verification) and backend/src/modules/storage/storage.service.ts (Supabase Storage uploads via `supabase.storage.from(bucket).upload(...)`).
- **Evidence, web/**: web/src/lib/auth.ts builds better-auth itself on `prismaAdapter(prisma)` — even web/'s own authentication system is Prisma-backed, not Supabase-Auth-backed, confirmed directly. web/src/lib/supabase/server.ts's `runWithTenantContext()` sets the DEC-001 `app.current_organization_id` session variable through a Prisma transaction, not a Supabase-client call. web/src/lib/db.ts constructs its runtime `PrismaClient` via the `@prisma/adapter-pg` driver-adapter pattern against a raw `pg.Pool` (web/package.json confirms `@prisma/adapter-pg` as a dependency) — Prisma is web/'s primary data-access layer too, matching backend/'s pattern.
- **Direct conflict with pre-restructure historical notes, recorded per this document's own rules**: .agents/context.md states, verbatim, "Tenant CRUD uses Supabase client SDK carrying authenticated JWTs; Prisma reserved for schema migrations and service-role cross-tenant admin operations," and .agents/tasks.md's Phase 1 checklist says essentially the same ("Establish Supabase Client SDK path for tenant CRUD & keep Prisma for schema/migrations/service-role queries"). Current code in both subsystems does the opposite: Prisma is the live domain-CRUD engine everywhere it is used; the Supabase client is a narrow, service-role-style singleton reserved for Supabase-specific capabilities. Per this document's stated rule, current code wins — this entry supersedes that portion of .agents/context.md and .agents/tasks.md for both subsystems.
- **Inferred rationale for keeping both rather than dropping one**: Supabase is not solely a Postgres host here — it also supplies (1) Supabase's Auth admin API, used as backend/'s fallback path for tokens that are not better-auth session tokens (backend/src/modules/auth/auth.repository.ts's `verifySupabaseJwtUser`), and (2) Supabase Storage, used for uploaded documents (backend/src/modules/storage/storage.service.ts; separately web/src/lib/supabase-storage.ts on the web/ side). Dropping Supabase would require replacing both; dropping Prisma would require hand-writing SQL for every domain query currently expressed as typed Prisma calls across both subsystems. Keeping both reads as pragmatic accretion — each tool doing what the other does not — rather than a deliberate, documented layering decision.
- **Known structural risk** (directly verified): backend/prisma/schema.prisma and web/prisma/schema.prisma are two independent files, confirmed byte-identical today (`diff` returns no differences), with no shared package, symlink, or codegen step tying them together, both pointed at the same conceptual Postgres database via separately-declared `DATABASE_URL` values (backend/.env.example, web/.env.example — though see DEC-003 regarding web/'s actual env files never defining this variable). Any future schema edit applied to one copy and not the other will silently desynchronize the two Prisma clients reading/writing the same physical tables — nothing in the repository (no CI, no shared schema package, no .github directory anywhere, confirmed by repository-wide search) would catch this.
- **Additional unresolved gap** (directly verified): backend/prisma/schema.prisma's datasource block declares no `url`; backend/ has no prisma.config.ts (only web/ has one, confirmed absent at backend/prisma.config.ts); backend/src/database/prisma.service.ts constructs `PrismaClient` with zero constructor arguments and backend/package.json does not depend on `@prisma/adapter-pg` at all (confirmed absent). web/ solves the identical "no inline datasource url" problem two ways (web/prisma.config.ts for the CLI; the adapter-pg pattern in web/src/lib/db.ts for runtime) that backend/ does not replicate. Whether backend/'s `PrismaService` can actually open a connection at runtime given this is Unknown — not verified by executing the app for this document.

### DEC-009: Two Independent npm Projects Instead of a Formal Monorepo Tool
- **Status**: INFERRED
- **Decision as observed**: backend/ and web/ are two entirely separate npm projects — each with its own package.json and package-lock.json, own tsconfig, own lint config, own test runner (Jest in backend/, confirmed via its package.json `"test": "jest"` script and embedded `"jest"` config block; Vitest in web/, confirmed via `"test": "vitest run"` and web/vitest.config.ts). No monorepo workspace tool is configured anywhere: no turbo.json, nx.json, pnpm-workspace.yaml, or lerna.json exists at the repo root or anywhere else (confirmed by direct filesystem search); there is no root-level package.json at all (confirmed absent), so there is no npm/yarn/pnpm `"workspaces"` field joining them either. The things that join the two subsystems are the repo-root docker-compose.yml (which, per DEC-003's current-code note, does not actually build or run either service correctly today), shared conventions (both TypeScript, both pointed at the same conceptual Postgres database, both holding a byte-identical copy of the same Prisma schema — see DEC-008), and one genuine, if fragile, code-level coupling: backend/src/config/config.module.ts's `ConfigModule.forRoot()` call sets `envFilePath: ['.env', '../web/.env']` — backend/'s NestJS config is written to also load web/.env as a fallback source, presuming backend/ and web/ sit as sibling directories, which in this repo's current layout they do — and backend/src/config/configuration.ts's `supabase.url` resolution falls back to `process.env.NEXT_PUBLIC_SUPABASE_URL`, a web/-side, browser-facing env var name, if `SUPABASE_URL` is unset (both confirmed by direct reading). Both mechanisms are uncommented and relative-path-dependent, and would silently break if either directory were renamed or moved — but they are real evidence that backend/'s config layer was written with a specific assumption about sitting next to web/, not built as a fully independent service.
- **Supporting evidence this is an early, not-yet-formalized state rather than a deliberate rejection of monorepo tooling**: git status shows web/, backend/, docker-compose.yml, and .agents/ as untracked in their entirety (see provenance note above for the complete, corrected list) — this whole split has never been committed, on any branch (git log --all -- web/ backend/ returns no history). There is no .github directory anywhere in the repository — no CI/CD pipeline exists to build, test, or coordinate either subsystem. There is no shared/, common/, packages/, or libs/ directory anywhere (confirmed by search) — the two subsystems do not currently share any code beyond the duplicated Prisma schema file, so a workspace tool's most common benefit (importing a shared internal package across projects) has nothing to operate on yet. (A related, minor data point: the root CONTRIBUTING.md still instructs contributors to `cd pawnify && npm install` and `cp .env.example .env` at the repository root — instructions that no longer resolve to anything, since there is no root package.json or root .env.example today; this is consistent with the overall picture of documentation not yet updated for the split.)
- **Inferred rationale**: backend/ appears to have been bootstrapped as an independent `nest new` project (backend/README.md is confirmed to be the fully unmodified default Nest scaffold) dropped alongside the pre-existing web/ app, with cross-cutting build tooling deferred until the split's shape stabilizes. Introducing a monorepo tool (Turborepo, Nx, pnpm workspaces) is ordinarily motivated by a need to share code or coordinate build/test graphs across packages; with essentially no shared code and zero CI today, that need has not yet visibly materialized — a plausible reason none has been introduced, but inferred from absence of evidence, not from a recorded decision to avoid monorepo tooling. It is equally possible one is planned but simply not yet implemented; the repository contains no evidence either way.
- **Cost already visible from not having one** (directly verified, cited under DEC-008): the two independent Prisma schema copies can already drift silently, and nothing in the repository would detect it.
