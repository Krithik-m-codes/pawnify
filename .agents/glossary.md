# Pawnify Glossary

## Scope & how to read this document

Pawnify is a multi-tenant pawn/asset-backed lending platform. The codebase is a monorepo mid-migration between two independent npm projects with no shared workspace tooling (no turborepo/nx/pnpm-workspace, no root package.json) and no CI (no .github, no other CI config found anywhere in the repo): backend/ is a new, partial NestJS API (only 7 modules exist: auth, cron, customers, health, market-rates, organizations, storage, webhooks); web/ is a mature Next.js 15 App Router application (Prisma, Supabase-hosted Postgres, better-auth, Redux Toolkit Query) that still contains almost all of the actual lending business logic (loans, payments, valuation, follow-ups, billing, settings, staff, reports, dashboard).

Every model and enum defined below lives in **one shared Prisma schema** — web/prisma/schema.prisma and backend/prisma/schema.prisma are byte-for-byte identical (verified by diff), as are web/prisma/seed.ts and backend/prisma/seed.ts. Both subsystems' Prisma layers are wired to read a `DATABASE_URL` environment variable at connection time (web/src/lib/db.ts; backend/src/config/configuration.ts) — neither schema.prisma's `datasource` block hardcodes a `url` at all, so the schema itself doesn't force one physical database over another. Whether the two subsystems actually point at the same physical database in a given deployment is an environment-configuration question the repository alone can't answer: in this checkout, backend/.env sets `DATABASE_URL`/`DIRECT_URL` to a Supabase pooler host, while web/.env, web/.env.example, and web/.env.prod define no `DATABASE_URL` or `DIRECT_URL` at all; the self-hosted docker-compose.yml path sets an identical `DATABASE_URL`/`DIRECT_URL` for its single "app" container, but that container's own build config (context ".", dockerfile "Dockerfile") points at a root-level Dockerfile that does not exist in this repository (only web/Dockerfile does) — which subsystem, if either, that self-hosted path actually builds is Unknown. Same-database connectivity for web/ specifically could not be confirmed from committed repository files alone. This means a model "existing" in the schema does **not** imply backend/ has any code that uses it — a repository-wide check found only four backend/ files that call Prisma against these models: backend/src/modules/customers/customers.repository.ts, backend/src/modules/organizations/organizations.repository.ts, backend/src/modules/market-rates/market-rates.repository.ts, and backend/src/modules/webhooks/webhooks.repository.ts (backend/src/modules/health/health.repository.ts also uses Prisma, but only for a raw `SELECT 1` connectivity check, not against any domain model). web/ has roughly 90 Prisma call sites (94 counted) across ~20 files covering the full business surface. Each entry below states explicitly which subsystem(s) the described behavior applies to.

All monetary and weight fields use Prisma's Decimal type (backed by decimal.js), never native JS floats — this is treated as a non-negotiable convention throughout web/src/lib/services.

Terms are grouped by domain area; enum values are listed under each enum entry. A "Drift" line marks a place where behavior actually observed in code diverges from what a name, comment, or the pre-restructure notes at .agents/context.md, .agents/decisions.md, .agents/lessons.md, .agents/schema.md, .agents/tasks.md would lead you to expect — those files describe the pre-extraction single-app design and predate backend/ entirely, so current code wins wherever they disagree. A final section collects the highest-value drift points in one place, and an appendix table gives a fast enum lookup.

## 1. Tenancy & Multi-Tenant Architecture

**Organization** — The tenant root of the entire schema. Fields: id, name, slug (unique), billingPlan (nullable string; null = self-hosted, or "starter"/"growth"/"enterprise"/"trial" for cloud — an informal convention, not a Prisma enum), createdAt, updatedAt. Almost every other domain and configuration table (Branch, LoanPolicy, DocumentType, Customer, Loan, LoanItem, LoanCharge, Payment, LedgerEntry, FollowUp, KycDocument) carries an organizationId foreign key that cascade-deletes when the Organization is deleted (web/prisma/schema.prisma). Both subsystems read/write Organization: web/'s billing service reads billingPlan for quota checks (web/src/lib/services/billing.ts), and backend/'s webhooks.repository.ts writes billingPlan directly (backend/src/modules/webhooks/webhooks.repository.ts).

**Branch** — A physical location/sub-unit of an Organization. Fields: id, organizationId (Cascade), name, address (nullable), createdAt, updatedAt. Staff (user) and Loan rows may optionally point at a Branch (SetNull on Branch deletion). Only one Branch index exists (@@index([organizationId])); there is no uniqueness constraint on (organizationId, name) (web/prisma/schema.prisma). web/'s billing.ts exposes checkCanCreateBranch to gate branch creation against a plan's maxBranches quota, but no caller of that function was found anywhere in web/src — branch creation is not actually quota-gated in practice despite the function existing (web/src/lib/services/billing.ts).

**Role** (enum, on the user model) — `PLATFORM_OPERATOR`, `OWNER`, `ADMIN`, `BRANCH_MANAGER`, `STAFF` (default). Defined once in the shared schema (web/prisma/schema.prisma). In practice, almost every privileged Server Action in web/ checks for the literal string `"ADMIN"` only (e.g. web/src/lib/auth/session.ts's checkAdmin/requireAdmin) — `BRANCH_MANAGER` and `PLATFORM_OPERATOR` are declared enum values with no observed authorization branch anywhere in web/ or backend/; `PLATFORM_OPERATOR` in particular has zero references outside schema.prisma in either subsystem. `OWNER` is the one exception: backend/'s organizations.controller.ts gates four endpoints (LoanPolicy update, team-member invite, team-member role change, team-member removal) with `@Roles('OWNER', 'ADMIN')` (backend/src/modules/auth/decorators/roles.decorator.ts, backend/src/modules/auth/guards/roles.guard.ts) — OWNER is treated as equal to, not distinct from, ADMIN there, but it's the only one of the three non-ADMIN/STAFF roles with any observed effect anywhere in the two subsystems. better-auth's user schema marks `role` as `input: false` (web/src/lib/auth.ts), so it can never be set by a public sign-up payload; staff provisioning is admin-gated. Separately, a platform-wide admin surface in web/ has no role gate at all — see Known Drift below.

**Row-Level Security (RLS) / Tenant Isolation** — A Postgres-layer defense that restricts which rows a database role can see/write based on a resolved organization id. Implemented in backend/sql/rls_policies.sql, which `ENABLE ROW LEVEL SECURITY`s on exactly 12 tables — Organization, Branch, LoanPolicy, DocumentType, Customer, KycDocument, Loan, LoanItem, LoanCharge, Payment, LedgerEntry, FollowUp — and defines a helper SQL function `current_user_organization_id()` that resolves, in order, a session-local Postgres setting (`app.current_organization_id`, via `current_setting(..., true)`) or, failing that, looks up the `organizationId` of the `"user"` row whose id equals `auth.uid()` (a Supabase-specific function). Organization itself gets only a `FOR SELECT` policy (no INSERT/UPDATE/DELETE policy at all); the other 11 tables get a `FOR ALL` policy comparing `"organizationId" = current_user_organization_id()` on both `USING` and `WITH CHECK`. This file is backend/-only (no equivalent exists anywhere under web/, aside from the unrelated test file discussed below) and is not wired into any Prisma migration or npm script in either subsystem (no prisma/migrations directory exists in this repository at all) — it must be applied to Postgres manually. No `FORCE ROW LEVEL SECURITY` is used, so the owning database role still bypasses all policies regardless of content (standard Postgres semantics). See Drift section below for enforcement-in-practice caveats.

**Session variable `app.current_organization_id`** — The custom Postgres session setting the RLS helper function reads first. The only code that sets it via `set_config(...)` is `runWithTenantContext` in web/src/lib/supabase/server.ts; a repository-wide search found no caller of that function anywhere else in web/src, and no occurrence of the setting at all under backend/src. Practically, nothing currently sets this value for any live request path in either subsystem, per the files reviewed.

## 2. Identity, Sessions & Staff Accounts

**user** (lowercase model name — better-auth convention) — The staff/account identity table, doubling as Pawnify's tenant-scoped "who is this person" record. Core better-auth fields: id (no @default — assigned externally by better-auth), name, email (unique), emailVerified, image. Domain fields layered on top (schema's own "Domain & Tenant fields" comment): role (Role, default `STAFF`), phone, isActive (default true), passwordHash (nullable), organizationId (nullable, SetNull), branchId (nullable, SetNull) (web/prisma/schema.prisma). Five named back-relations record staff attribution elsewhere: `customersCreated` ("CreatedByStaff"), `loansHandled` ("HandledByStaff"), `paymentsCollected` ("CollectedByStaff"), `followUpsAssigned` ("AssignedStaff"), `kycVerified` ("VerifiedByStaff"). Despite carrying the same organizationId/branchId tenant-scoping shape as every RLS-protected domain table, `user` is **not** in rls_policies.sql's `ENABLE ROW LEVEL SECURITY` list at all (backend/sql/rls_policies.sql) — this table has no RLS policy of any kind.

**session / account / verification** — The remaining better-auth tables (web/prisma/schema.prisma, section commented "BETTER AUTH TABLES"). `session` holds a unique `token`, `expiresAt`, and a required `userId` (Cascade); it has no index on `userId` at all (only the unique `token`). `account` holds one linked credential/OAuth account per provider (`providerId`, `accountId`, optional `password` for the email/password provider); it has no compound unique on (`providerId`, `accountId`) and no index on `userId`. `verification` holds email-verification / password-reset tokens keyed by `identifier`; unusually, its `createdAt`/`updatedAt` are nullable-with-default (every other model's are non-null), and it has no relation to `user` at all (looked up by `identifier`) and no index of any kind, including none on `identifier` itself. None of these three carry an `organizationId`, so they sit outside the RLS scheme entirely — and, like `user`, are not covered by any RLS policy despite holding session tokens and password hashes.

**better-auth** — The self-hosted authentication library web/ actually uses (`betterAuth(...)` in web/src/lib/auth.ts, backed by `prismaAdapter(prisma, { provider: "postgresql" })`), not Supabase Auth directly. web/'s own session-checking layer (web/src/lib/auth/session.ts — `getSession`, `requireSession`, `checkAuth`, `checkAdmin`) calls exclusively `auth.api.getSession(...)`; it never falls back to a Supabase JWT. Supabase in this repo is used for hosting the Postgres database (and RLS, and Storage), not as web/'s primary sign-in mechanism.

**Dual-path token validation (backend/)** — backend/'s `AuthService.validateToken` (backend/src/modules/auth/auth.service.ts) first tries to resolve a better-auth session token by looking it up in the `session` table (backend/src/modules/auth/auth.repository.ts's `findSessionByToken`); only if that fails does it fall back to verifying the token as a Supabase JWT (`verifySupabaseJwtUser`, via `supabase.auth.getUser(token)`). So backend/ accepts either a better-auth session token or a Supabase-issued JWT — it does not mint or verify its own independently-signed JWT_SECRET token despite `JWT_SECRET` existing in backend/.env.example (no `jsonwebtoken`, `@nestjs/jwt`, or any `jwt.sign`/`jwt.verify` call was found anywhere in backend/src). backend/'s `SupabaseService` (backend/src/database/supabase.service.ts) itself instantiates its Supabase client preferring the service-role key (falling back to the anon key, then to an empty string) — the service-role key path bypasses RLS by Supabase design — and is otherwise used for raw SQL (`query()`) and Storage bucket operations, not for scoping ordinary Prisma queries to a caller's identity.

## 3. Configuration & Lending Policy

**LoanPolicy** — The per-organization rulebook: a 1:1 row keyed by a unique `organizationId` (Cascade). Fields: `currencyCode` (default `"USD"`, ISO 4217), `currencySymbol` (default `"$"`), `weightUnit` (WeightUnit, default `GRAM`), `purityExpression` (PurityExpression, default `KARAT`), `dayCountConvention` (DayCountConvention, default `ACTUAL_365`), `gracePeriodDays` (default 7), `mandatoryIdThreshold` (Decimal(12,2), default 0 — "0 = disabled" per the schema comment), `mandatoryIdDocTypeId` (nullable string — **no `@relation` declared**, despite the name implying a foreign key to DocumentType; its target is unenforced), `ltvTiers` (required Json — an ascending array of `{ maxValue: number|null, ltvPercent: number }`, shape not enforced by the database) (web/prisma/schema.prisma). Populated at organization onboarding by web/src/app/(app)/onboarding/onboarding-wizard-client.tsx, which offers per-jurisdiction presets (see Jurisdiction Profile below).
Drift: `weightUnit`, `purityExpression`, and `dayCountConvention` are captured here but not read back anywhere in web/'s own loan-creation or interest-accrual call sites — see the Known Drift section.
Drift (backend/): backend/'s `organizations.repository.ts` (`getOrganizationPolicy`) returns a hand-built fallback object when no LoanPolicy row yet exists for an organization, defaulting `currencyCode`/`currencySymbol` to `"INR"`/`"₹"` — disagreeing with the Prisma schema's own `"USD"`/`"$"` default. That same fallback object also fabricates flattened fields (`ltvTier1Percent`, `ltvTier2Percent`, `ltvTier3Percent`, `ltvTier1Max`, `ltvTier2Max`, `defaultInterestMonthly`, `safetyMarginPercent`) that do not exist as actual LoanPolicy columns — so a caller of this method gets a materially different shape depending on whether a LoanPolicy row already exists (backend/src/modules/organizations/organizations.repository.ts).

**DocumentType** — A per-organization, admin-configurable identity-document type, meant to replace a fixed KYC enum with something operators can define themselves (e.g. "Passport", "Aadhaar", "PAN", "SSN", "Driver's License", per the schema's own comment). Fields: `organizationId` (Cascade), `name`, `isPrimaryId` (default false — marks a document type as acceptable for the mandatory-ID check), `maskingRule` (nullable, e.g. `"last4"`) (web/prisma/schema.prisma). `KycDocument.documentTypeId` optionally links to it (SetNull). Nothing prevents more than one DocumentType per organization from having `isPrimaryId = true` — web/prisma/seed.ts's demo data deliberately seeds two ("PAN Card" and "Passport") simultaneously marked primary.

**AppSetting** — The single genuinely global (non-tenant) table: `key` (String, primary key) → `value` (String). No `organizationId`, no timestamps, no relations (web/prisma/schema.prisma). Used as a flat key-value store for values that overlap conceptually with LoanPolicy but are shared across every organization on a deployment: LTV tier boundaries/percentages (`ltv.tier1.max`, `ltv.tier1.percent`, `ltv.tier2.max`, `ltv.tier2.percent`, `ltv.tier3.percent`), `interest.default.monthly`, `grace.period.days`, `pan.threshold`, plus live market-rate cache keys (`rate.gold.per_gram`, `rate.silver.per_gram`, `rate.last_updated`) and `valuation.safety.margin`. Read/written by web/src/lib/services/settings.ts (generic CRUD; only `updateSetting` has any caller elsewhere in web/src — `getSetting`, `getSettings`, `updateSettings`, and `groupSettings` have none) and by web/src/lib/services/market-rates.ts, and independently by backend/src/modules/market-rates/market-rates.repository.ts. Neither file explains why this global table and the per-organization LoanPolicy both encode overlapping settings (web/prisma/seed.ts seeds both with the same numbers: 85/80/75% tiers, 7-day grace, 50,000 PAN threshold — though seed.ts notably does not seed the `rate.gold.per_gram`/`rate.silver.per_gram`/`rate.last_updated`/`valuation.safety.margin` keys, so a freshly seeded deployment starts on fallback metal rates until a live refresh runs — see Market Rates below).

**WeightUnit** (enum, on LoanPolicy and consumed as an optional parameter of the valuation pipeline) — Values: `GRAM` (default), `TROY_OUNCE`, `TOLA`. Conversion to grams happens in `convertToGrams` (web/src/lib/services/valuation.ts): 1 Troy Ounce = 31.1034768 g; 1 Tola = 11.6638038 g (the file's comment specifies this as "Standard South Asian / Middle Eastern 3/8 oz tola"); `GRAM` passes through unchanged. All conversions round to 4 decimal places.
Drift: `LoanItemInput`/`loanItemSchema` (web/src/lib/services/loans.ts, web/src/lib/validation/loan.ts) carry no `weightUnit` field at all, so `createLoan` always calls `computeItemValuation` with the `GRAM` default — `TROY_OUNCE`/`TOLA` conversion is reachable only by calling `convertToGrams` directly, not through the actual loan-creation path.

**PurityExpression** (enum, on LoanPolicy) — Values: `KARAT` (default), `MILLESIMAL_FINENESS`, `PERCENTAGE`. A conversion helper `normalizePurityPercent` exists in web/src/lib/services/valuation.ts: `KARAT` → `value / 24 × 100` (interprets the input as a 0–24 karat number); `MILLESIMAL_FINENESS` → `value / 10` (interprets the input as a 0–1000 fineness number, e.g. 916 → 91.6%); `PERCENTAGE` → passthrough.
Drift: `normalizePurityPercent` has no call site anywhere in web/src, including inside `computeItemValuation` itself, which consumes `purityPercent` as an already-normalized raw percentage. `LoanPolicy.purityExpression` therefore has no code path that actually applies it.

**DayCountConvention** (enum, on LoanPolicy) — Values: `ACTUAL_365` (default), `ACTUAL_360`, `THIRTY_360`. Governs the divisor used to annualize a monthly interest rate into a daily rate (web/src/lib/services/interest.ts's `computeDailyInterest`: `annualRate = monthlyRate × 12`; divisor is `360` only if the convention is exactly `ACTUAL_360`, else `365`).
Drift: the divisor logic only distinguishes two buckets — `THIRTY_360` receives no distinct treatment (no 30-day-month calendar logic exists anywhere) and is computed identically to `ACTUAL_365`. Separately, `Loan` itself has no `dayCountConvention` column, and neither `loans.ts` (`getLoanById`, `closeLoan`) nor `payments.ts` (`recordPayment`, `previewPaymentAllocation`) ever supplies a convention override when computing interest — every actual interest computation in these services falls back to `ACTUAL_365` regardless of the organization's configured `LoanPolicy.dayCountConvention`.

**Jurisdiction Profile** — A static, non-persisted reference registry (`WORLDWIDE_JURISDICTIONS`, web/src/lib/config/jurisdictions.ts) of 12 institutional presets, each carrying a currency, locale, default day-count convention, a KYC document list, and a one-line regulatory description. `getJurisdictionProfile(code)` case-normalizes and falls back to `GLOBAL` if unrecognized (or if no code is given at all). Consumed by the onboarding wizard (to pre-fill a new organization's LoanPolicy) and by a public marketing calculator component; `getJurisdictionProfile`/`WORLDWIDE_JURISDICTIONS` themselves have no other call site in web/src beyond their own Vitest test.

**Asset Category** — A second static, non-persisted registry (`UNIVERSAL_ASSET_CATEGORIES`, web/src/lib/config/asset-categories.ts) modeling 6 collateral classes beyond precious metals, each with a `defaultLtvPercent`, example appraisal fields, and example items (e.g. Rolex Submariner under `LUXURY_WATCHES`, Hermès Birkin under `LUXURY_GOODS`). `getAssetCategory(id)` falls back to `GENERAL_COLLATERAL` if unrecognized.
Drift: this registry is consumed only by a public marketing-site calculator component and its own Vitest test — it is not referenced anywhere in the actual loan-creation code path, which supports only `MetalType.GOLD`/`SILVER` items (see MetalType below). The 6 categories' `defaultLtvPercent` values span 50–80%; only `PRECIOUS_METALS`' 80% coincides with one of the 85/80/75% LTV tiers actually used by `valuation.ts` (the middle tier) — the other five (50–65%) don't match any of the three.

## 4. Customers & KYC

**Customer** — A borrower record. Fields: `organizationId` (Cascade), `fullName`, `phone`, `email` (nullable), `dob` (nullable), `addressLine1`, `addressLine2` (nullable), `city`, `state`, `pincode`, `photoUrl` (nullable), `createdById` (required, → user, "CreatedByStaff") (web/prisma/schema.prisma). `@@unique([organizationId, phone])` — a phone number is unique per organization, not globally. web/'s customers.ts (web/src/lib/services/customers.ts) implements create/list/typeahead-search/detail lookups; `searchCustomers` requires at least 2 query characters and matches `fullName`/`phone`. backend/'s own `CustomersRepository.searchCustomers` (backend/src/modules/customers/customers.repository.ts) implements the same fullName/phone search **but with no `organizationId` filter in its `where` clause at all** — its cross-tenant isolation, if any, depends entirely on RLS, whose enforcement-in-practice is itself in question (see Row-Level Security above and Known Drift below). A Customer can be hard-deleted via web/'s admin-gated `deleteCustomerAction` (web/src/app/(app)/customers/[id]/actions.ts) as long as it has no currently-`ACTIVE` loan; deletion cascades to manually deleting that customer's KycDocuments, LoanItems, Payments, LoanCharges, LedgerEntries, FollowUps, and Loans first (see LedgerEntry drift note below).

**KycDocument** — One identity document evidencing a Customer's identity. Fields: `organizationId` (Cascade), `customerId` (Cascade), `docType` (KycDocType, default `CUSTOM`), `documentTypeId` (nullable → DocumentType, SetNull), `docNumber` (no uniqueness constraint at any scope), `fileUrl` (nullable), `status` (KycStatus, default `PENDING`), `verifiedById` (nullable → user, "VerifiedByStaff") (web/prisma/schema.prisma). Has `createdAt` but **no `updatedAt`**, so a status transition (e.g. `PENDING` → `VERIFIED`) leaves no timestamp trail. `updateKycStatus` (web/src/lib/services/customers.ts) sets `verifiedById` to the acting staff member whenever the new status is anything other than `PENDING`, and clears it back to null if status reverts to `PENDING`.

**KycDocType** (enum) — Values: `AADHAAR`, `PAN`, `VOTER_ID`, `PASSPORT`, `DRIVING_LICENSE`, `CUSTOM` (default). `CUSTOM` is meant to pair with an organization's own DocumentType rows.
Drift: web/'s Zod validation schema `kycDocumentSchema` (web/src/lib/validation/customer.ts, whose own header comment calls it "the source of truth for validation") only accepts 5 of the 6 values (`AADHAAR`, `PAN`, `VOTER_ID`, `PASSPORT`, `DRIVING_LICENSE`) — a `CUSTOM`-typed document cannot pass this schema as written, even though it is the Prisma column's own default value.

**KycStatus** (enum) — Values: `PENDING` (default), `VERIFIED`, `REJECTED`.

**Mandatory ID Threshold** (a.k.a. "PAN threshold" in code/comments) — The rule that a customer's total loan exposure crossing a configured amount requires a verified primary identity document before a new loan can be disbursed. Implemented by `checkPanRequired(customerId, newPrincipalAmount?, organizationId?)` (web/src/lib/services/customers.ts). Threshold resolution: if `organizationId` is given, use that org's `LoanPolicy.mandatoryIdThreshold` (Decimal, "0 = disabled") and that org's `DocumentType` rows with `isPrimaryId = true` as the accepted primary-document list (replacing, not merging with, the default `["PAN"]`); otherwise fall back to the global `AppSetting` key `pan.threshold` (default 50000). "Required" = `threshold > 0 AND (sum of the customer's ACTIVE loan principals + the proposed new principal) >= threshold`. A document counts as satisfying the requirement if its `docType` is exactly `PAN`, or it links to a DocumentType with `isPrimaryId = true`, or its `docType.toString()` matches a primary-document name.
Drift: one of the four OR-conditions in the "has document" check — the third, sitting between the `DocumentType.isPrimaryId` check and the final `docType.toString()` check — compares the document's **`docNumber`** (not its `docType`) against the primary-document-type names (`primaryDocTypeNames.includes(doc.docNumber.toUpperCase())`). This compares the wrong field and is unlikely to ever match in practice. `createLoan` (web/src/lib/services/loans.ts) calls `checkPanRequired` and throws if required-but-missing, blocking loan creation.

**maskDocNumber** — A display helper (web/src/lib/validation/customer.ts) that replaces every character of a document number except the last 4 with a bullet character ("•"), for PII-safe rendering in the UI. Returns the input unchanged if it is 4 characters or shorter.

## 5. Loans & Collateral Valuation

**Loan** — The core lending contract. Key fields (web/prisma/schema.prisma): `organizationId` (Cascade), `branchId` (nullable, SetNull), `loanNumber` (e.g. `PL-2026-000001`), `customerId`, `handledById` (→ user, "HandledByStaff"), `loanDate`, `dueDate`, `tenureMonths`, `interestRateMonthly` (Decimal(6,3)), `ltvPercent` (Decimal(5,2)), `gracePeriodDays` (default 7 — a per-loan snapshot, independent of `LoanPolicy.gracePeriodDays`), `totalAssessedValue`/`principalAmount`/`principalOutstanding` (all Decimal(12,2)), `lastSettledDate` (the interest-accrual anchor date), `status` (LoanStatus, default `ACTIVE`), `closedAt`/`closedById` (both nullable — `closedById` is a bare string with **no `@relation`**, unlike every other staff-attribution field on this model), `notes`. `@@unique([organizationId, loanNumber])`. Created via web/'s `createLoan` (web/src/lib/services/loans.ts), which recomputes every item's valuation server-side, enforces the billing-plan active-loan quota, enforces the mandatory-ID gate, enforces the LTV-derived principal cap, and runs inside a Serializable-isolation transaction (`runSerializable`, web/src/lib/db.ts) to keep loan-number generation collision-free under concurrency. Can be hard-deleted via `deleteLoanAction` (web/src/app/(app)/loans/[id]/actions.ts), admin-gated, which manually deletes all child LoanItems/Payments/LoanCharges/LedgerEntries/FollowUps before deleting the Loan row itself.

**LoanItem** — One piece of pledged collateral within a Loan. Fields: `organizationId` (Cascade), `loanId` (Cascade), `metalType` (MetalType), `description`, `purityLabel` (freeform display string, e.g. "22K" — no format enforcement against the `PURITY_PRESETS` reference table), `purityPercent` (Decimal(5,2)), `grossWeightGrams`/`stoneWeightGrams`/`netWeightGrams`/`fineWeightGrams` (all Decimal(10,4)), `valuationRatePerGram` (Decimal(10,2)), `assessedValue` (Decimal(12,2)), `packetNumber`, `storageLocation`, `photoUrl` (nullable), `releasedAt` (nullable — set when the physical item is handed back) (web/prisma/schema.prisma). `@@unique([organizationId, packetNumber])`. Has `createdAt` but no `updatedAt`.

**LoanStatus** (enum) — Values: `ACTIVE` (default), `CLOSED`. **There is no `OVERDUE` value in the enum.** "Overdue" is a purely derived, never-persisted display status computed by `deriveLoanDisplayStatus` (web/src/lib/services/loans.ts): a `CLOSED` loan is always `"CLOSED"`; otherwise, if today is past `dueDate + gracePeriodDays`, the displayed status is `"OVERDUE"`, else `"ACTIVE"`. The source comment explicitly frames this as deliberate ("avoids needing a scheduled job and can never drift out of sync with the calendar"). Because this filter can't be expressed as a database WHERE clause, `getLoans` must fetch every DB-`ACTIVE` loan and paginate the `ACTIVE`/`OVERDUE` split in application memory when that specific filter is requested (web/src/lib/services/loans.ts).

**MetalType** (enum) — Values: `GOLD`, `SILVER`. This is the **only** collateral typing actually enforced by the loan-creation pipeline (`loanItemSchema`, web/src/lib/validation/loan.ts, matches this enum exactly) — the broader "Asset Category" marketing registry (watches, art, vehicles, luxury goods) described above is not wired into any enforced validation.

**Gross weight / stone weight / net weight / fine weight** — The item-valuation pipeline (web/src/lib/services/valuation.ts, `computeItemValuation`): `grossWeightGrams` is the item's total weight as weighed; `stoneWeightGrams` (default 0) is the weight of non-metal inclusions (gems, etc.) to be deducted; `netWeightGrams = grossWeightGrams − stoneWeightGrams` (throws if negative — "stone weight exceeds gross weight"); `fineWeightGrams = netWeightGrams × (purityPercent / 100)` — the pure-metal-equivalent weight. `assessedValue = round(fineWeightGrams × valuationRatePerGram, 2)`, where the rate is documented as always being a pure (999) metal rate. `createLoan` always recomputes every item's valuation server-side from these raw inputs — it never trusts a client-submitted `assessedValue`/`netWeight`/`fineWeight` (web/src/lib/services/loans.ts, citing this as "Non-Negotiable #5").
Cross-field note: `loanItemSchema` (web/src/lib/validation/loan.ts) additionally requires `stoneWeightGrams` to be strictly less than `grossWeightGrams` — stricter than `computeNetWeight`, which would itself tolerate the two being equal (yielding exactly zero net weight, since it only rejects a negative result).

**LTV (Loan-to-Value)** — The percentage of a collateral's assessed value that may be advanced as loan principal. `getLtvPercent(totalAssessedValue, slabs)` (web/src/lib/services/valuation.ts) scans a tiered slab table in the order given and returns the first slab whose `maxValue` is ≥ the total assessed value (the code depends on slabs being pre-sorted ascending; it does not sort them itself). `computeEligibleAmount = totalAssessedValue × (ltvPercent / 100)`, rounded to 2 decimal places. `createLoan` throws if the requested `principalAmount` exceeds this eligible amount.

**LTV Tiers / Slabs** — Resolved by `getLtvSlabs(organizationId?)` (web/src/lib/services/valuation.ts) in priority order: (1) the organization's `LoanPolicy.ltvTiers` JSON array, if non-empty (a tier's `maxValue` of `null` or the literal string `"Infinity"` marks the open-ended top tier); (2) else, if all five relevant `AppSetting` keys (`ltv.tier1.max`, `ltv.tier1.percent`, `ltv.tier2.max`, `ltv.tier2.percent`, `ltv.tier3.percent`) are present, a 3-tier table built from them; (3) else the hardcoded default:

| Assessed value | LTV % |
|---|---|
| ≤ 250,000 | 85% |
| ≤ 500,000 | 80% |
| > 500,000 (unbounded) | 75% |

A partially-configured `AppSetting` set (some but not all five keys present) falls through entirely to the hardcoded default rather than partially applying (web/src/lib/services/valuation.ts).

**Principal Amount / Principal Outstanding / Total Assessed Value** — `principalAmount` is the amount disbursed at loan creation (must be positive and ≤ the LTV-eligible amount); `principalOutstanding` starts equal to `principalAmount` and is reduced by each payment's allocated-principal portion; `totalAssessedValue` is the sum of all recomputed item `assessedValue`s at creation time (web/src/lib/services/loans.ts, web/prisma/schema.prisma).

## 6. Interest, Charges, Payments & Ledger

**Interest accrual** — Simple (non-compounding) interest computed on read, never persisted as a running balance (web/src/lib/services/interest.ts). `computeDailyInterest = principalOutstanding × (monthlyRate × 12) / divisor / 100` (divisor per DayCountConvention, see above). `computeAccruedInterest(loan, asOfDate)` = daily interest × calendar days since `lastSettledDate` (via date-fns `differenceInCalendarDays`), rounded to 2 decimal places, or 0 if non-positive. `computeMonthlyInterest = principalOutstanding × (monthlyRate / 100)` is a separate, day-count-independent display figure ("this month's interest").

**LoanCharge** — A one-off fee or penalty attached to a Loan. Fields: `organizationId` (Cascade), `loanId` (Cascade), `chargeType` (ChargeType), `amount` (Decimal(12,2)), `isSettled` (default false) (web/prisma/schema.prisma). Has `createdAt` but no `updatedAt` (toggling `isSettled` leaves no timestamp).
Drift: among the code reviewed, only `PROCESSING_FEE` charges are ever created (`createLoan`, web/src/lib/services/loans.ts, only if a positive `processingFee` is supplied) — no code path creates a `PENAL_CHARGE` or `OTHER` charge, and the derived `OVERDUE` display status does not itself trigger any charge or rate change.

**ChargeType** (enum) — Values: `PROCESSING_FEE`, `PENAL_CHARGE`, `OTHER`.

**Payment** — One repayment event. Fields: `organizationId` (Cascade), `loanId` (no explicit `onDelete`), `receiptNumber` (e.g. `REC-20260706-00001`), `paymentDate`, `amountPaid` (Decimal(12,2)), `mode` (PaymentMode), `allocatedCharges`/`allocatedInterest`/`allocatedPrincipal` (all Decimal(12,2), default 0), `collectedById` (→ user, "CollectedByStaff"), `notes` (nullable) (web/prisma/schema.prisma). `@@unique([organizationId, receiptNumber])`. Has `createdAt` only — treated as an immutable event by convention, though nothing in the schema or RLS SQL enforces that at the database level (no CHECK constraint ties `amountPaid` to the sum of the three allocated fields either). Created by `recordPayment` (web/src/lib/services/payments.ts). Can be hard-deleted alongside its parent Loan via the admin-gated `deleteLoanAction`/`deleteCustomerAction` described above — see the Known Drift note on LedgerEntry immutability.

**PaymentMode** (enum) — Values: `CASH`, `UPI`, `BANK_TRANSFER`, `CARD`.

**Repayment Waterfall** (a.k.a. payment allocation waterfall) — The fixed order in which a single payment amount is applied, implemented inside one atomic `runSerializable` transaction by `recordPayment` and previewed read-only (no transaction) by `previewPaymentAllocation` (both web/src/lib/services/payments.ts):
1. **Charges** — outstanding LoanCharge rows, oldest-first by `createdAt`, each capped at `min(remaining, charge.amount)`; a charge is marked `isSettled = true` once fully paid.
2. **Interest** — accrued interest as of the payment date (via `computeAccruedInterest`), capped at `min(remaining, accruedInterest)`.
3. **Principal** — capped at `min(remaining, principalOutstanding)`.

If more than a 0.01 rounding tolerance remains unallocated after all three steps, the entire transaction throws and nothing persists — **overpayment is rejected outright**, never partially applied or silently dropped. `lastSettledDate` (the interest clock) advances fully to the payment date only if accrued interest was paid in full; on a partial interest payment it instead advances by a proportional fraction of the elapsed days, specifically to avoid silently forgiving unpaid interest. Loan closure (`closeLoan`) requires zero outstanding principal (strict, no tolerance), zero unsettled charges, and accrued interest not exceeding a 0.01 tolerance; item release (`releaseItems`) is a separate, later step gated on `status === CLOSED` — closure is a financial event, physical hand-back of collateral is a distinct operational one (web/src/lib/services/loans.ts).

**LedgerEntry** — An append-style financial audit-trail row. Fields: `organizationId` (Cascade), `loanId` (no explicit `onDelete`), `type` (TransactionType), `amount`/`principalAfter` (both Decimal(12,2) — `principalAfter` is a running-balance snapshot), `referenceId` (nullable string, **no `@relation`** — target model not stated anywhere, Unknown), `description` (freeform string; ledger descriptions in web/'s code hardcode the ₹ symbol regardless of the organization's configured `currencySymbol`) (web/prisma/schema.prisma). Has `createdAt` only, and only a single 3-column composite index (`@@index([organizationId, loanId, createdAt])`), unlike the separate single-column indexes used elsewhere in the schema.
Drift: .agents/context.md describes this as an "immutable... financial audit trail," but neither the schema nor backend/sql/rls_policies.sql contains any trigger or `REVOKE UPDATE/DELETE` that would enforce that — and web/'s admin-gated `deleteLoanAction`/`deleteCustomerAction` (web/src/app/(app)/loans/[id]/actions.ts, web/src/app/(app)/customers/[id]/actions.ts) explicitly hard-delete LedgerEntry rows along with their parent Loan/Customer. Immutability here is a convention observed by the *creation* code paths, not an enforced constraint.

**TransactionType** (enum, on LedgerEntry) — Values: `DISBURSEMENT`, `PAYMENT`, `CLOSURE`, `ITEM_RELEASE`. One `DISBURSEMENT` entry is written at loan creation, one `PAYMENT` entry per recorded payment, one `CLOSURE` entry (amount 0) at closure, one `ITEM_RELEASE` entry (amount 0) at item release.

**Loan Number / Receipt Number generation** — `generateLoanNumber` produces `PL-{4-digit year}-{6-digit zero-padded sequence}` (sequence = count of the org's existing loans starting with `PL-{year}`, plus 1); `generateReceiptNumber` produces `REC-{YYYYMMDD}-{5-digit zero-padded sequence}` (sequence = count of the org's payments created since local midnight, plus 1) (web/src/lib/services/loans.ts, web/src/lib/services/payments.ts). Neither is backed by a dedicated unique-sequence column — collision-avoidance instead relies entirely on both being generated only inside a `runSerializable` Serializable-isolation transaction (web/src/lib/db.ts), which causes Postgres to abort one of two concurrent conflicting transactions with a serialization failure (error code `P2034`) that is automatically retried up to 3 times.

## 8. Billing & SaaS Plan Tiers

**billingPlan** — A nullable free-text field on Organization (not a Prisma enum) whose informal convention is: `null` = self-hosted; `"starter"` | `"growth"` | `"enterprise"` | `"trial"` = a cloud subscription tier (web/prisma/schema.prisma comment). Resolved to a concrete quota config by `getPlanConfig` (web/src/lib/services/billing.ts), which treats `null` or any unrecognized string as `self_hosted` (i.e. unlimited).

**Plan tiers** (web/src/lib/services/billing.ts, `PLAN_QUOTAS`):

| Plan key | Display name | Max active loans | Max branches | Price / mo (USD) |
|---|---|---|---|---|
| self_hosted (billingPlan = null or unrecognized) | Self-Hosted Community | ∞ | ∞ | $0 |
| starter | Cloud Starter | 100 | 1 | $49 |
| growth | Cloud Growth | 1,000 | 5 | $149 |
| enterprise | Cloud Enterprise | ∞ | ∞ | $399 |

**checkCanCreateLoan / checkCanCreateBranch** — Quota-enforcement gates (web/src/lib/services/billing.ts). `checkCanCreateLoan(organizationId)` counts `ACTIVE` loans for the org and throws if the count already meets the plan's `maxActiveLoans`; it runs at the very top of `createLoan` (web/src/lib/services/loans.ts), before any transaction, making it a hard gate. `checkCanCreateBranch` implements the identical pattern against `maxBranches`, but has no caller anywhere in web/src — branch creation is not actually quota-gated in the current code despite the function's existence. `updateOrganizationPlan` (web/src/lib/services/billing.ts) simply updates `Organization.billingPlan`; backend/'s `WebhooksRepository.updateOrganizationPlan` (backend/src/modules/webhooks/webhooks.repository.ts) does the same update directly via Prisma (`updateMany`) — a second, independent write path to the same field. It is driven by a `POST /webhooks/dodo-payments` endpoint (backend/src/modules/webhooks/webhooks.controller.ts, webhooks.service.ts) that maps Dodo Payments' `subscription.active`/`subscription.renewed` events to `newPlan = metadata.planId` (default `"starter"`) and `subscription.cancelled`/`subscription.expired`/`subscription.failed` events to `newPlan = null`; any other event type is silently ignored. That endpoint carries no `JwtAuthGuard` (expected, since a third-party webhook caller has no user session), but it does verify a real HMAC-SHA256 signature: it binds the body via `@Req() req: RawBodyRequest<Request>` (raw-body capture enabled by `NestFactory.create(AppModule, { rawBody: true })` in backend/src/main.ts, needed because signature verification requires the exact original bytes) and passes `req.rawBody` plus the request headers to the official `dodopayments` npm SDK's `client.webhooks.unwrap()`, which implements the Standard Webhooks spec (via its `standardwebhooks` dependency) and throws on an invalid or missing signature — the service catches that and returns a real `401 Invalid webhook signature` response. It still checks that `DODO_PAYMENTS_API_KEY`/`DODO_PAYMENTS_WEBHOOK_KEY` are configured first, returning 501 if not. See Known Drift below.

## 9. Market Rates & Scheduled Refresh

**Market Rate** — The cached spot price used to value gold/silver collateral (`goldRatePerGram`, `silverRatePerGram`, per-gram, always for 24K/fine metal). Stored in the global `AppSetting` table under keys `rate.gold.per_gram`, `rate.silver.per_gram`, `rate.last_updated`, alongside cached copies of the LTV tiers, PAN threshold, and interest/grace defaults (web/src/lib/services/market-rates.ts, backend/src/modules/market-rates/market-rates.repository.ts). Because `AppSetting` has no `organizationId` column, this rate is shared across every organization on a deployment, not per-tenant.
Drift: this feature is implemented **twice, independently**: web/src/lib/services/market-rates.ts (`fetchAndSaveLiveMetalRates`, invocable only via an ADMIN-gated Server Action, web/src/app/(app)/admin/settings/actions.ts) and backend/src/modules/market-rates/market-rates.service.ts (same function, exposed at `GET`/`POST /cron/update-rates`, guarded by `CronAuthGuard` checking a `Bearer {CRON_SECRET}` header — backend/src/modules/cron/cron.controller.ts, backend/src/modules/cron/guards/cron-auth.guard.ts; if `CRON_SECRET` itself is unset, the guard allows every request through). Both hit the same third-party spot-price APIs (goldprice.org primary, open.er-api.com + gold-api.com secondary) using the same 1 troy oz = 31.1034768 g conversion, and write the same global AppSetting keys. Their fallback numbers mostly, but not entirely, agree: web/'s `DEFAULT_RATES` hardcodes `ltvTier1/2/3Percent`=85/80/75, `ltvTier1/2Max`=250000/500000, `defaultInterestMonthly`=1.5, `defaultGraceDays`=7, and `panThreshold`=50000 — all matching the literal fallback strings in backend/'s `getConfiguredDefaults()` (e.g. `process.env.LTV_TIER1_PERCENT || '85.0'`). The two disagree specifically on the metal rates: web/ hardcodes `goldRatePerGram`=7850.0/`silverRatePerGram`=98.5, while backend/ sources these from `DEFAULT_GOLD_RATE_PER_GRAM`/`DEFAULT_SILVER_RATE_PER_GRAM` environment variables that appear in neither backend/.env nor backend/.env.example — so backend/'s fallback for these two fields is 0/0 whenever no cached AppSetting row exists yet (e.g. immediately after a fresh seed, since web/prisma/seed.ts never seeds `rate.gold.per_gram`/`rate.silver.per_gram`). This asymmetry aside, the shared API integration and AppSetting keys still indicate the live-refresh responsibility was migrated toward backend/ without web/'s own cron declaration being removed or its route restored (see next entry).

**Cron-triggered rate refresh** — web/vercel.json declares its own daily Vercel Cron Job (`"schedule": "0 0 * * *"`, i.e. midnight) hitting path `/api/cron/update-rates` — but **no route file exists at web/src/app/api/cron/update-rates** (confirmed by enumerating every `route.ts` under web/src/app: the only one present is web/src/app/api/auth/[...all]/route.ts, the better-auth catch-all). web/'s declared cron would therefore 404 if Vercel ever invoked it. backend/'s `/cron/update-rates` endpoint is the one with an actual working handler.

## 10. Precious-Metal & Pawn-Industry Terminology

**Karat** — A gold-purity scale out of 24 parts (24K = ~99.9% pure). Used as the default `PurityExpression` on `LoanPolicy` and reflected in the reference table `PURITY_PRESETS.GOLD` (web/src/lib/services/valuation.ts): 24K → 99.9% (fineness 999), 22K → 91.6% (fineness 916), 18K → 75.0% (fineness 750), 14K → 58.5% (fineness 585). This preset table has no consumer anywhere else in web/src by static search — it is defined but not currently wired into any item-entry UI or validation path found in this review.

**Millesimal Fineness** — A purity scale expressed in parts per 1000 (e.g. 999 = 99.9% pure, 916 = 91.6% pure). One of the three `PurityExpression` enum values; `normalizePurityPercent`'s `MILLESIMAL_FINENESS` branch divides the input by 10 to get a percentage (web/src/lib/services/valuation.ts) — see the Drift note under PurityExpression above regarding this helper never actually being called.

**Tola** — A traditional South Asian / Middle Eastern unit of weight for gold/silver, defined in this codebase as 11.6638038 grams (the "3/8 oz tola" variant, per the source comment) (web/src/lib/services/valuation.ts, `convertToGrams`). One of the three `WeightUnit` enum values.

**Troy Ounce** — The international unit of weight for precious metals, defined here as 31.1034768 grams. Used both as a `WeightUnit` enum value for item weights and as the standard unit that live gold/silver spot-price APIs quote in, requiring conversion (`rate / 31.1034768`) to get a per-gram rate (web/src/lib/services/valuation.ts; web/src/lib/services/market-rates.ts; backend/src/modules/market-rates/market-rates.service.ts).

**Purity / Fineness** — The proportion of actual precious metal in an item, expressed as `purityPercent` (Decimal(5,2), 0–100) on `LoanItem`. Combined with net weight to derive `fineWeightGrams` (the pure-metal-equivalent weight an item's assessed value is actually based on) — see "Gross weight / stone weight / net weight / fine weight" above.

**Hallmark** — A purity-certification mark/stamp on gold jewelry (in India, typically a BIS Hallmark). In this codebase the word appears only as UI copy — a photo-upload label "Hallmark / Purity Photo" in web/src/components/document-uploader.tsx and cosmetic marketing copy ("Hallmark Badge", web/src/components/marketing/navbar.tsx) — there is no `Hallmark` model, field, or enum value anywhere in web/prisma/schema.prisma. It is not a modeled or enforced concept; it is documentary/marketing language only.

**Assay** — The technical process of testing a metal sample to determine its purity. Like Hallmark, this term appears only as marketing/decorative copy in the codebase (e.g. "ASSAY CERTIFIED" badge text in web/src/components/marketing/footer.tsx, a "Decorative Assay Watermark" comment in web/src/components/marketing/interactive-ltv-calculator.tsx) — there is no modeled assay/certification entity, field, or workflow anywhere in the schema or services reviewed. Purity in this system is a self-reported `purityPercent` input recomputed server-side, not the output of a modeled assay process.

**AUM (Assets Under Management)** — Industry shorthand, used directly in code (`totalAUM`, web/src/lib/services/dashboard.ts) for the sum of `principalOutstanding` across all `ACTIVE`-bucketed loans (both truly `ACTIVE` and derived-`OVERDUE`). Displayed on the dashboard as "Total Active AUM" (web/src/app/(app)/dashboard/dashboard-client.tsx) and recomputed independently for reporting (`totalActiveAUM`, web/src/app/(app)/reports/actions.ts, which does not import or call `deriveLoanDisplayStatus` and instead reimplements the grace-period bucketing inline) — the two are separately coded, not shared through one function.

## 11. Known Spec-vs-Code Drift — Quick Reference

The pre-restructure notes at .agents/context.md and .agents/decisions.md describe a design intent from before backend/ existed, and lessons.md itself flags spec-vs-code drift as a recurring risk in this project (its own "Code vs. Spec Drift Findings" section). The most load-bearing gaps found while compiling this glossary:

- **RLS "verified" claim vs. actual test**: .agents/decisions.md (DEC-001) labels tenant isolation "load-bearing" and cites its verification as "Tested via `src/__tests__/rls-isolation.test.ts`." That file (web/src/__tests__/rls-isolation.test.ts) only filters plain in-memory JavaScript arrays — it never connects to Postgres or exercises backend/sql/rls_policies.sql. Whether RLS is actually enforced for any live connection (which depends on the connecting Postgres role's `BYPASSRLS`/ownership status — Unknown from the repository alone) was not established by that test.
- **user table has no RLS at all**: despite carrying the same `organizationId`/`branchId` shape as every protected domain table, `user` is absent from backend/sql/rls_policies.sql's `ENABLE ROW LEVEL SECURITY` list entirely (see Row-Level Security above).
- **Platform-operator page and action have no role gate**: web/src/app/(app)/platform-admin/page.tsx calls `checkAuth()` and discards the result without checking `.authenticated` or any role — it renders every organization's billing plan, active-loan count, branch count, and user count for any session that reaches it. The only gate in front of it is the shared `(app)` route group's layout (web/src/app/(app)/layout.tsx), which calls `requireSession()` and only requires a non-empty `role` and `isActive: true` — not any specific role. Its billing-plan-change action (`changeOrganizationPlanAction`, web/src/app/(app)/platform-admin/actions.ts) is gated by comparing the caller's email to a `PLATFORM_ADMIN_EMAIL` environment variable that appears in none of web/.env, web/.env.example, or web/.env.prod; when that variable is unset, the gate's `if (adminEmail && ...)` condition short-circuits false and the check is skipped entirely — so in every environment configuration visible in this repository, any authenticated staff member of any Role, from any organization, can view and change any other organization's billing plan.
- **Webhook signature verification (RESOLVED — billing webhook migrated from Stripe to Dodo Payments)**: backend/'s billing webhook used to be `POST /webhooks/stripe` and accepted unsigned requests (no guard; the handler only checked that `STRIPE_WEBHOOK_SECRET` was non-empty before trusting the POST body directly). The Stripe integration has since been fully replaced with Dodo Payments: the route is now `POST /webhooks/dodo-payments` (backend/src/modules/webhooks/webhooks.controller.ts), and the handler (backend/src/modules/webhooks/webhooks.service.ts) now verifies a real HMAC-SHA256 signature via the `dodopayments` SDK's `client.webhooks.unwrap(rawBody, { headers })` before trusting `type`/`data.metadata.{organizationId, planId}` to decide which organization's `billingPlan` to overwrite — `unwrap()` throws on an invalid/missing signature, which the service turns into a `401 Invalid webhook signature` response. This was verified working end-to-end: an unsigned curl POST against the running dev server returned `401 {"error":"Invalid webhook signature"}`. The endpoint still 501s if `DODO_PAYMENTS_API_KEY`/`DODO_PAYMENTS_WEBHOOK_KEY` is unset.
- **"Immutable" ledger vs. real hard-delete paths**: LedgerEntry and Payment are framed as an immutable audit trail in .agents/context.md, but web/'s admin-gated `deleteLoanAction` and `deleteCustomerAction` (web/src/app/(app)/loans/[id]/actions.ts, web/src/app/(app)/customers/[id]/actions.ts) hard-delete both, and nothing in the schema or SQL blocks it.
- **LoanPolicy currency default disagreement**: schema.prisma defaults `currencyCode`/`currencySymbol` to `"USD"`/`"$"`; backend/'s organizations.repository.ts fallback (used pre-provisioning) defaults to `"INR"`/`"₹"`, and also fabricates several flattened tier fields that aren't real LoanPolicy columns.
- **Configured-but-unread policy fields**: `LoanPolicy.weightUnit`, `purityExpression`, and `dayCountConvention` are captured at onboarding but not read back by web/'s own `createLoan`/interest-accrual call sites — every loan is actually processed as if `weightUnit=GRAM`, purity is a raw already-normalized percentage, and `dayCountConvention=ACTUAL_365`, regardless of what the organization configured.
- **Duplicated, partially-broken market-rate cron**: web/ and backend/ each independently re-implement the same live gold/silver rate fetch against the same global AppSetting keys, with slightly different fallback numbers for the metal rates themselves (see Market Rates above); web/vercel.json's own declared daily cron route has no matching handler in web/src/app, so only backend/'s `/cron/update-rates` endpoint currently works.
- **Hardcoded ₹ symbol despite multi-currency schema**: `LoanPolicy` carries a configurable `currencyCode`/`currencySymbol`, and config/jurisdictions.ts models 12 currencies, but web/'s LedgerEntry descriptions and several thrown-error messages (web/src/lib/services/loans.ts, web/src/lib/services/payments.ts) hardcode the ₹ glyph regardless of the organization's configured currency.
- **THIRTY_360 is declared but unimplemented**: the enum value exists, but `computeDailyInterest`'s divisor logic only distinguishes `ACTUAL_360` from everything else — `THIRTY_360` is computed identically to `ACTUAL_365`.
- **Mandatory-ID check compares the wrong field on one branch**: `checkPanRequired`'s third OR-condition compares a KYC document's `docNumber` (not its `docType`) against primary-document-type names (web/src/lib/services/customers.ts).
- **backend/'s team-member creation bypasses better-auth's signup flow**: `organizations.repository.ts`'s `addTeamMember` creates `user` rows directly via `prisma.user.upsert` with a custom id scheme (`user_${Date.now()}_${random}`), unlike web/ and web/prisma/seed.ts, which exclusively provision users through `auth.api.signUpEmail`. No corresponding `account` (credential) row is created by this path, so whether a team member added this way could authenticate via better-auth's email/password login is Unknown from the repository alone.
